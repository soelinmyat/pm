## Pressure-Test

**Goal:** Find the weakest points in the emerging direction before the user commits to it. A direction that survives pressure-testing is worth grooming; one that doesn't saves weeks of wasted effort.

**How:**

Stress-test the direction across these dimensions:
- **Assumptions:** What are we assuming that might not be true?
- **Risks:** What could go wrong?
- **Open questions:** What do we still not know?
- **Dependencies:** What has to be true first?

Don't list all of these — surface the 2-3 that actually matter for *this* idea. Prioritize the ones that would kill the idea if wrong, not the ones that would merely complicate it.

Present your sharpest concern first as a direct question:
> "The biggest risk I see is [X]. If [condition], this falls apart. How confident are you that [condition] holds?"

This beat can be multiple exchanges. Push back if the user hand-waves a real risk, but accept "we'll figure it out" for manageable unknowns. The goal is informed confidence, not certainty.

**Done-when:** The user has addressed or acknowledged the top risks. You can summarize the direction as "we're going with X, despite Y, because Z" — if you can't fill in all three slots, keep pushing.

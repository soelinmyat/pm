## Reframe

**Goal:** Shift the conversation from the user's initial framing to the underlying problem or a higher-leverage angle. This is the most valuable beat — a good reframe changes what gets built.

**How:**

Apply one of these reframing lenses (pick the one that creates the biggest gap between the user's current framing and a better one):

**Jobs to Be Done:** Strip the solution away. What's the job the user is hiring this product to do? A job has the form: *"When [situation], I want to [motivation], so I can [outcome]."* If the user described a feature, reframe as the job. The best reframes reveal that the user's feature solves the wrong job, or the right job but for the wrong situation.

**Problem vs. Solution:** If the user described a solution, push toward the problem. If they described a problem, push toward a sharper version of it — who has this problem most acutely, and what's the cost of the status quo?

**Must-have test:** Would the target user be genuinely disappointed if this didn't exist? If the answer is "they'd find a workaround," the framing isn't sharp enough. Push toward the version where the answer is "they'd be stuck."

**Simpler framing:** Is there a version of this idea that captures 80% of the value with 20% of the complexity? Often the first framing is overbuilt because it tries to solve every edge case.

If the framing is already sharp, say so explicitly and move on — the evaluation is mandatory, but the conclusion can be "the current framing holds."

Share your reframe as a short, opinionated take:

> "The way I'd reframe this: [reframe]. The real unlock is [insight]."

Then ask if the reframe resonates or if you're off base. One question.

**Done-when:** The user either confirms the reframe resonates, offers their own reframe, or says the original framing is correct. You now have a shared problem statement to explore approaches against.

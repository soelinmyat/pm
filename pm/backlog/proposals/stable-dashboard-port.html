! Problem & Context

Every time a user launches the PM dashboard, it binds to a random ephemeral port (49152-65535). Bookmarks break, browser tabs become stale, and users must re-copy the URL from the terminal. This undermines PM's key architectural differentiator: a local, browser-accessible dashboard that no competitor offers.

Research signal: Every major dev tool (Vite, Next.js, Storybook, Jupyter) uses a fixed, memorable port. None randomize. The current PM behavior is an anti-pattern that falls in the OS ephemeral port range, creating additional collision risk with OS-allocated outbound connections on macOS and Windows.

▢ Scope Overview

✓ In Scope

  • Hash project directory path to deterministic port in 3000-9999 range
  • Auto-increment on collision (port+1, +2, ...) with user-visible message
  • Preserve PM_PORT env var as explicit override
  • Print assigned port in server-started JSON (already works)

✗ Out of Scope

  • Port persistence to file (hash is deterministic from path)
  • Port reservation/locking (over-engineering for single-user tool)
  • Port configuration UI (env var override sufficient)
  • Host/bind address changes (separate concern)
Gap-fill — table stakes that's currently broken

☰ Issue Breakdown

PM-038 Stable dashboard port via project-path hashing

Users can bookmark and return to the PM dashboard without reconfiguring it after each session.

infrastructure dx medium priority
  1. Port is derived deterministically from the absolute project directory path using a hash mapped to the 3000-9999 range.
  2. If the hashed port is occupied, auto-increment until a free port is found. Log: "Port {hashed} occupied, using {actual} instead."
  3. PM_PORT env var overrides the hash when set (existing behavior preserved).
  4. start-server.sh always passes CALLER_DIR as PM_PROJECT_DIR to server.js.
  5. Port resolution happens inside startServer(), not at module scope (async promotion required).
  6. Both server.listen() callsites use the async-resolved port.
  7. Fallback to process.cwd() when PM_PROJECT_DIR is unset (direct node server.js only).
  8. Server-started JSON "url" field format: "http://localhost:{PORT}" (must not regress).
  9. Unit tests for hashProjectPort() (determinism + range) and port probe (collision fallback). Existing tests pass unchanged.

? Open Questions

Questions the decision-maker should be prepared to discuss:

  • Why hash-per-project rather than a single fixed port like every other dev tool? (Answer: multi-project simultaneous use — each project gets its own stable URL without manual config.)
  • If a user's hashed port is consistently occupied by another dev server (e.g., Next.js on 3000), their "stable" bookmark is permanently shifted. Is a deny-list of popular ports worth adding?
  • What is the integration test coverage for the async startServer() promotion? Unit tests cover the hash and probe in isolation, but the wired behavior in both listen callsites has no automated integration test.

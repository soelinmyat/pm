! Problem & Context

The PM dashboard currently treats all sections (research, strategy, backlog) as equal peers with flat navigation — burying the proposal artifact that represents PM's highest-quality output. The groom workflow is PM's core capability, but the dashboard doesn't reflect this. Users who complete a groom session have no natural place to revisit, browse, or share their proposals.

Research signal: Productboard places "initiatives" as the hero artifact between strategy and tasks. Dashboard UX research consistently recommends 5–7 key elements per view with hierarchical navigation. The current flat structure with 5 equal tabs violates both principles. The user's vision: dashboard as a visual companion to the CLI, open on the left while the agent works on the right.

This initiative reshapes the dashboard to reflect PM's actual workflow hierarchy: groom/proposals as the hero, research/strategy/competitors as supporting reference material.

▢ Scope Overview

✓ In Scope

  • Proposal metadata sidecar (phase-5.8 writes JSON alongside each proposal HTML)
  • Active session indicator on dashboard home (topic, phase, date from .groom-state.md)
  • Proposal gallery as home page (card grid with gradients, badges, issue counts)
  • Restructured navigation: Home → Proposals → Backlog → Knowledge Base
  • Backlog grouped by proposal (with flat kanban as alternate view)
  • Proposal detail view (embedded HTML via iframe in dashboard frame)

✗ Out of Scope

  • Real-time WebSocket groom phase streaming (deferred — current WS infra supports reload, not granular state)
  • Companion mode groom integration (deferred — phase-by-phase visual updates is a separate initiative)
  • Proposal status workflow (in-review → shipped) (deferred — requires backlog schema changes)
  • Standalone backlog kanban removal (flat kanban stays as alternate view)

☰ Issue Breakdown

PM-025 Dashboard Proposal-Centric Redesign

Users open the PM dashboard and immediately see their proposals as the hero artifact — active groom session at top, proposal gallery as centerpiece, backlog grouped by proposal, knowledge base as reference shelf.

dashboard ux groom high priority
  1. Home page shows active groom session indicator when .pm/.groom-state.md exists.
  2. Home page displays proposal card gallery with completed and draft proposals.
  3. Navigation restructured to Home → Proposals → Backlog → Knowledge Base.
  4. Knowledge Base contains Research, Competitors, and Strategy as sub-tabs.
  5. Backlog default view groups issues by parent proposal with toggle to flat kanban.
  6. Proposal detail view embeds proposal HTML within dashboard frame via iframe.
  7. phase-5.8 writes proposal-meta.json sidecar alongside each proposal HTML.
  8. All existing dashboard URLs continue to resolve (redirects acceptable).
  9. Empty states render gracefully when no proposals or groom session exist.
PM-026 Proposal Metadata Sidecar

The dashboard can reliably read proposal metadata (title, date, verdict, phase, issue count) without parsing freeform HTML.

infrastructure groom high priority
  1. phase-5.8 writes {slug}.meta.json alongside each {slug}.html.
  2. JSON schema: title, date, verdict, verdictLabel, phase, issueCount, gradient, labels.
  3. Defines shared helpers: readProposalMeta(slug) and readGroomState(pmDir).
  4. Gradient assigned deterministically from slug hash (palette of 8).
  5. Legacy proposals without sidecar degrade gracefully (title from filename, no badge).
  6. Draft proposals rendered from groom state fields only: topic, phase, started.
PM-027 Active Session Indicator

Users see their current groom session status immediately on the dashboard home page — topic name, human-readable phase, and start date.

dashboard ux high priority
  1. Reads .pm/.groom-state.md via shared readGroomState() helper.
  2. Banner shows exactly 3 fields: topic name, human-readable phase, start date.
  3. Phase mapping: intake → "Intake", research → "Research", scope → "Scoping", etc.
  4. When no groom state exists: no banner, gallery renders at top.
  5. Corrupted state file: silently omitted, no error page.
  6. Banner is read-only — no links, no controls, all plain text.
PM-028 Proposal Gallery Page

Users can browse all their proposals — completed and in-progress — as a visual card gallery, making the proposal the hero artifact.

dashboard ux high priority
  1. New /proposals route scans *.meta.json files.
  2. Cards show hero gradient, title, verdict badge, issue count, date.
  3. Draft cards: dashed border, not clickable, current phase badge.
  4. Completed cards click through to /proposals/{slug} detail view.
  5. Home shows 6 most recent; /proposals shows all.
  6. Empty state with /pm:groom hint when no proposals exist.
  7. Directory existence checked per request, not at startup.
PM-029 Navigation Restructure — KB Umbrella

Dashboard navigation reflects PM's workflow hierarchy: Home → Proposals → Backlog → Knowledge Base.

dashboard ux medium priority
  1. navLinks array updated to 4 items: Home, Proposals, Backlog, KB.
  2. KB page with server-side tab activation via ?tab= query param.
  3. Old URLs redirect via HTTP 302: /research → /kb?tab=research, etc.
  4. Competitor detail and research topic pages continue to work.
  5. Active nav highlighting specified per URL pattern.
PM-030 Backlog Grouped by Proposal

Users see their backlog issues organized under their parent proposal — giving every issue context about why it exists.

dashboard backlog medium priority
  1. Default view: issues grouped by parent proposal (matched via .meta.json lookup).
  2. Parent chain walk: if parent points to an issue, walk up until proposal ancestor found.
  3. Proposal groups show gradient swatch, title, status breakdown.
  4. "Standalone Issues" section for items with no proposal ancestor.
  5. View toggle: "By Proposal" (default) | "Kanban" via ?view= param.
  6. Flat kanban renders identically to current implementation.
PM-031 Proposal Detail View

Users can view a full proposal embedded within the dashboard frame — with back navigation and standalone link.

dashboard ux medium priority
  1. New /proposals/{slug} route with two-step path traversal prevention.
  2. Proposal HTML in iframe, embed header reads "PROPOSAL" (not "WIREFRAME").
  3. Fixed 800px iframe height (dynamic sizing deferred).
  4. "Back to Proposals" breadcrumb + "Open standalone" link.
  5. 404 with back link if proposal file doesn't exist.
  6. Sticky TOC verified to not overlap dashboard chrome.

? Open Questions

Questions the decision-maker should be prepared to discuss:

  • The 10x claim is relative to the current flat dashboard — is this about better user orientation, or better output quality? These are different arguments that need to be separated for stakeholders.
  • What does "empty state" look like for a new PM user who has never run /pm:groom? The home page would show no proposals and no active session. Is this more or less welcoming than the current stats overview?
  • How do we measure success without product analytics (strategy non-goal #3)? Post-ship plan: community survey, changelog feedback thread, and tracking when PM-023 (public dashboard) provides an analytics surface.
  • The active session banner is the highest-value item but never updates in real-time (groom state is outside the file watcher). Is this acceptable, or does it undermine the "visual companion" vision?
  • How should the gallery handle proposals with multiple bar-raiser iterations? Current answer: sidecar is overwritten with latest verdict. Historical verdicts are not tracked (deferred with status workflow).

! Problem & Context

PM has 17 command files that do nothing but delegate to skills — they are redundant architecture.

The using-pm skill, preloaded at session start via SessionStart hook, already routes every user request to the correct skill. Commands are a second invocation path that duplicates the same routing. This creates maintenance burden (17 extra files to keep in sync) and architectural inconsistency with the ecosystem trend toward skills-only plugins.

Research signal: Analysis of 18 plugins in the ecosystem shows skills-dominant is the mature pattern. Superpowers v5.0.5 (the most widely adopted plugin) has deprecated all commands in favor of skill-only architecture with SessionStart hook bootstrap.

▢ Scope Overview

Delete commands, preserve inline logic, update manifests and docs.

✓ In Scope

  • Delete all 17 command files from commands/
  • Verify using-pm skill routing covers all command workflows
  • Update plugin.json manifests (Claude + Cursor)
  • Update docs/README for skill-only invocation

✗ Out of Scope

Gap-fill — architectural cleanup, not differentiation

? Open Questions

Four questions for the decision-maker before approving.

  • How do new users discover what PM can do without tab-completable commands? (The using-pm routing table in the session context replaces the command list, but it's invisible to the user. The README becomes the primary discovery surface.)
  • Should PM-055 (groom-centric-messaging) ship first to establish groom-first positioning, then be adapted for skill-only? Or should this initiative absorb PM-055's intent into the docs rewrite?
  • Does Cursor handle skill-only plugins identically to Claude Code? (Superpowers works on both, suggesting yes, but PM should verify after merging.)
  • What is the rollback plan if skill-only activation has unexpected failures? (Commands can be restored from git history, but the real safety net is the smoke test in PM-057 AC5.)

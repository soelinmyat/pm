! Problem & Context

The strategy doc has all the right content but can't be shared as a story.

Product engineers write structured strategy documents (ICP, positioning, priorities, non-goals) but when they need to explain their strategy to a co-founder, advisor, or first hire, the reference-style format doesn't work. They end up manually rebuilding the content in Keynote, Google Slides, or Notion — losing time and context.

Research signal: No PM tool generates strategy as a narrative slide deck. Standalone presentation tools (Gamma, Beautiful.ai, Chronicle) generate decks but have zero PM workflow integration — no access to research, landscape, or competitor data. The McKinsey SCR framework (Situation → Complication → Resolution) maps directly to PM's three data sources.

▢ Scope Overview

Synthesize the full knowledge base into a narrative slide deck with consulting-grade structure.

✓ In Scope

  • HTML slide deck template with SCR narrative flow (~10 slides)
  • Synthesize strategy.md + landscape.md + competitors/ into one story
  • Keyboard nav, full-screen, slide counter, progress dots
  • Visual elements: positioning map, key players table, market stats
  • Action titles that assert claims, not label topics
  • Dark/light mode, graceful degradation for missing sources
  • Strategy skill update + on-demand regeneration via /pm:strategy deck

✗ Out of Scope

  • Speaker notes / presenter view (low value for 1-person use)
  • PDF/PPTX export (browser print-to-PDF works)
  • Slide transitions / animations (keeps it clean and fast)
  • Editable slides in-browser (generated artifact, not an editor)
  • Custom theming beyond dark/light (premature)
⚡ 10x — no competitor synthesizes a multi-source knowledge base into a narrative deck

PM-064 Strategy Narrative Slide Deck

Product engineers can explain their strategy to anyone by generating a narrative slide deck directly from their PM knowledge base.

strategy visual-companion medium priority
Show acceptance criteria (7)
  1. Running the strategy skill with visual_companion: true offers to generate a narrative slide deck. /pm:strategy deck regenerates on demand without re-running the interview.
  2. Generated deck is a self-contained HTML file at pm/strategy-deck.html — opens in any browser, no dependencies.
  3. Deck follows an SCR narrative arc with action titles that tell the full story when read in sequence.
  4. Arrow keys navigate between slides. Full-screen button and slide counter present.
  5. Dark/light mode adapts to system preference.
  6. Deck synthesizes data from all available sources: strategy.md (always), landscape.md (if exists), competitors/ (if exists).
  7. When sources are missing, the deck gracefully produces a shorter but coherent presentation.
PM-065 Strategy deck template + skill integration

Product engineers get a working narrative slide deck from strategy.md alone — the minimum viable strategy presentation.

strategy template medium priority
Show acceptance criteria (12)
  1. New template at templates/strategy-deck.html with placeholder slots for each slide.
  2. Each slide occupies exactly one viewport height (100vh). No scrolling within slides.
  3. Left/right arrow keys navigate. Slide counter visible (e.g., "3 / 7").
  4. Full-screen button uses Fullscreen API.
  5. Progress dots indicate current position.
  6. Dark/light mode via CSS variables from scripts/frame-template.html.
  7. 7-slide SCR narrative from strategy.md: Title → ICP → Problem → Differentiation → Positioning → Priorities → Metrics.
  8. Action titles assert specific claims. Topic labels fail (e.g., "Market overview" fails, "No tool covers the full lifecycle" passes).
  9. SKILL.md updated: deck offered when visual_companion: true, second offer after positioning map. On-demand via /pm:strategy deck. If strategy.md missing, exits with message.
  10. Named placeholder tokens defined as interface contract for PM-066.
  11. Generated file written to pm/strategy-deck.html and opened in browser.
  12. HTML file fully self-contained — no external dependencies, works offline.
PM-066 Knowledge base synthesis into strategy deck

The strategy deck tells the full product story by weaving in market landscape data and competitive intelligence.

strategy research medium priority
Show acceptance criteria (10)
  1. When landscape.md exists: market stats slide from <!-- stat: --> comments, key players slide (max 6 rows). Skips gracefully if comments absent.
  2. When competitors/ has profiles: competitive landscape slide with key gaps from index.md or matrix.md.
  3. When positioning map data exists: 2x2 plot slide. Reproduces CSS from strategy-canvas.html, omits interactive click behavior.
  4. Full deck follows SCR arc (~10 slides): SITUATION (market + players + ICP) → COMPLICATION (problem + gaps + whitespace) → RESOLUTION (differentiation + priorities + metrics).
  5. Missing landscape.md: skips market slides, remains coherent ~7-slide deck.
  6. Missing competitors/: skips competitive slides.
  7. Both missing: falls back to strategy.md-only (PM-065 baseline).
  8. Slide count and progress dots dynamically adjust.
  9. Action titles on data-enriched slides reference specific numbers or findings.
  10. Data provenance footer on enriched slides: source name, count, and date from updated: frontmatter.

? Open Questions

Four questions for the decision-maker before approving.

  • This is off-priority. Why build it now? (Low-cost quick win: one template + one SKILL.md update leveraging existing infrastructure. Ships alongside groom visual companion work using the same config flag.)
  • Who is the user presenting to, and how often? (Co-founders, advisors, first hires. Frequency varies — the on-demand regeneration path supports both one-time and recurring use.)
  • How do we ensure action title quality? (PM-065 AC8 defines a claim-vs-topic test with examples. The SKILL.md update includes prompting guidance. Quality is best-effort with a clear pass/fail criterion.)
  • What happens if this fails? (Low blast radius: one template file, one SKILL.md section. No impact on core strategy or groom workflows. Can be removed without affecting existing functionality.)

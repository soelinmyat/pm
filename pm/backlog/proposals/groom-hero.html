! Problem & Context

A 4-step linear prerequisite chain (setup → research → strategy → groom) suppresses grooming adoption and kills first-session trust.

PM currently requires users to run /pm:setup, then /pm:research, then /pm:strategy before they can run /pm:groom — the command that produces the most tangible value. This sequential pipeline contradicts every competitor's approach and violates progressive disclosure principles: NN/g recommends revealing complexity at the moment of readiness, not before. Users who install PM expecting to groom an idea hit a wall of prerequisite commands and either abandon or take a suboptimal shortcut.

Research signal: No competing AI coding plugin requires a linear setup pipeline before the user can do real work. Evil Martians (2026) identifies "works out of the box" as a primary trust signal for developer tool adoption — a user hitting a setup gate before their first real command is a trust failure, not just a friction point. cc-sdd's /kiro:spec-init bootstraps everything from a single command.

PM-052 Groom-Centric Entry Point

Users install PM and start grooming immediately — no linear prerequisite chain, no setup command, no mandatory strategy step before doing real work.

epic onboarding workflow high priority
Show acceptance criteria (8)
  1. A brand new project with no .pm/ directory can run /pm:groom <idea> and reach Phase 1 without errors — no setup-gate message or prompt to run /pm:setup is shown.
  2. A brand new project can run /pm:research <topic> without prior setup — same zero-friction guarantee.
  3. Users who have no strategy can unblock grooming inline without running the full strategy skill (see PM-054).
  4. README "Get Started" section leads with /pm:groom <idea> as the first command.
  5. using-pm/SKILL.md positions groom and research as primary entry points.
  6. SessionStart hook setup check is a hint, not a warning.
  7. Dashboard home page shows a prominent "Start Grooming" CTA when no proposals/sessions exist.
  8. Dashboard restructured with proposals and groom sessions as primary, research/strategy as reference.
PM-053 Auto-bootstrap config on first groom/research

Users run /pm:groom or /pm:research on a fresh project and it just works — no prior /pm:setup required.

onboarding workflow high priority
Show acceptance criteria (7)
  1. When /pm:groom starts and .pm/config.json does not exist, groom Phase 1 creates .pm/config.json with the nested schema matching setup.
  2. When /pm:research starts and .pm/config.json does not exist, research creates the same default config before proceeding.
  3. If .pm/ directory exists but config.json does not (partial state), the bootstrap creates the file without clobbering existing contents.
  4. If .pm/config.json already exists, the bootstrap is a no-op — no overwriting, no merging. If malformed JSON, groom warns and proceeds with in-memory defaults.
  5. The bootstrap creates .pm/ directory if it doesn't exist. Groom's guard creates .pm/groom-sessions/.
  6. The pm/ knowledge base directory is also created if it doesn't exist.
  7. No error message, warning, or prompt to run /pm:setup is shown to a user whose first command is /pm:groom or /pm:research.
PM-054 Quick-start inline strategy during groom

Users who have never written a strategy can unblock groom Phase 2 with 3 questions instead of running the full strategy skill.

workflow onboarding high priority
Show acceptance criteria (8)
  1. Groom Phase 2, when pm/strategy.md is missing, presents three options: (a) full strategy skill, (b) quick-start (3 questions), (c) skip.
  2. Quick-start asks exactly 3 questions, one at a time: (1) Who is your target user? (2) Top 3 priorities? (3) What are you explicitly NOT building?
  3. Quick-start writes pm/strategy.md with frontmatter and Sections 1 (derived product identity), 2 (ICP), 6 (priorities), and 7 (non-goals).
  4. Sections 3 (value prop) and 4 (competitive positioning) are written as explicit stubs: "Not yet defined — run /pm:strategy to expand."
  5. The minimal strategy passes the Phase 2 strategy gate: file exists, frontmatter has type: strategy, Sections 2/6/7 are present and non-empty.
  6. Scope review agents encountering stub Sections 3/4 produce a review noting sections are pending — no errors, review states competitive positioning was not evaluated.
  7. A groom session using quick-start strategy references the ICP answer in target audience context — strategy grounds the output, not just the gate.
  8. Quick-start flow completes in under 2 minutes of user interaction time (3 questions, no follow-ups).

? Open Questions

Four questions surfaced during bar raiser review for the decision-maker.

  • What is the expected quick-start strategy adoption rate vs. full strategy? If most users pick quick-start, Sections 3/4 stubs become the norm and scope review quality degrades across the board.
  • How should the fresh-project first-run experience differ from the existing-project experience? Currently the CTA logic has two distinct empty states (pm/ missing vs pm/ with no content) — is the distinction visible enough?
  • What heuristic should suggestedNext use to determine when to stop recommending /pm:groom and start surfacing other commands? After N sessions? After N proposals?
  • What grooming rate target makes this initiative a success? Measure: percentage of issues that go through groom before entering dev. Current baseline is unknown — instrument before shipping.

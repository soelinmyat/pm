! Problem & Context

Multi-phase groom sessions produce dense terminal output that is hard to scan, review, and reference.

Product engineers running groom sessions scroll through terminal history to find earlier phase outputs, re-read state files manually, and lose context as sessions grow. The terminal is the right medium for conversation, but not for presenting structured content like scope grids, review verdicts, and issue breakdowns.

Research signal: No AI PM competitor offers a terminal+browser split for grooming. Dev tools like Playwright (--ui), Storybook, and Vite prove the pattern works. Evil Martians research shows multi-phase CLI processes need phase steppers with checkmarks, not raw text.

▢ Scope Overview

Seven capabilities in scope; companion deprecation and background cleanup deferred.

✓ In Scope

  • Opt-in prompt at groom session start
  • /session/{slug} route in dashboard server
  • Per-phase HTML generation (phases 4–5.8)
  • Placeholder screen for non-visual phases
  • Multiple concurrent sessions
  • Session cleanup in Phase 6
  • Dashboard home shows active sessions with links

✗ Out of Scope

  • Interactive browser UI (scope creep)
  • Mobile-responsive layouts (wrong ICP)
  • Accumulated scroll view (complexity)
  • Custom themes (premature)
  • Companion mode deprecation (separate follow-on)
  • 24h background cleanup (Phase 6 handles it)
★ 10x — No competitor offers this

⚙ Technical Feasibility

Feasible with caveats — builds heavily on existing dashboard infrastructure.

✓ Build On

  • routeDashboard() URL dispatch chain
  • watchDirectoryTree() + WebSocket live-reload
  • readGroomState() parses groom sessions
  • GROOM_PHASE_LABELS for phase stepper
  • renderMarkdown() inline renderer
  • .pm/config.json visual_companion key

★ Build New

  • /session/{slug} route handler
  • Watcher extension for .pm/sessions/
  • Per-phase HTML write steps (6 phase files)
  • Companion screen template
  • Opt-in prompt in Phase 1

⚠ Risks

  • WebSocket broadcast reloads all tabs (v1 tradeoff)
  • LLM-authored HTML quality varies
  • Namespace collision with companion mode (mitigated: groom- prefix)

➤ Sequencing

  • 1. PM-060: Route + opt-in (MVP)
  • 2. PM-061: Phase screens (enrichment)
  • 3. PM-062: Dashboard links + cleanup

☰ Issue Breakdown

One parent issue with three sequential children — MVP first, enrichment second, polish third.

PM-036Groom Session Visual Companion
Users who opt in see each grooming phase rendered as a clean, scannable web page — the terminal stays lightweight for conversation while the browser carries the substance.
Highoutput-qualityfeature
Acceptance Criteria (8)
  1. Skill checks .pm/config.json for visual_companion after Phase 1; asks once if unset, persists answer.
  2. Dashboard server serves /session/{slug} route showing current phase as formatted HTML.
  3. Phases 4–5.8 write current.html summarizing phase output.
  4. Non-visual phases show "Phase in progress..." placeholder.
  5. Multiple concurrent sessions supported at their own URLs.
  6. Dashboard home shows active sessions with companion links.
  7. Phase 6 cleans up session directory after issue creation.
  8. .pm/sessions/ is gitignored.
PM-060Session route and opt-in prompt
Users who opt in at groom start see a live browser page at /session/{slug} that shows the current phase state.
Highinfrastructure
Acceptance Criteria (9)
  1. New /session/{slug} route in routeDashboard(); reads groom state, renders as HTML.
  2. If .pm/sessions/groom-{slug}/current.html exists, serves that instead.
  3. Unknown slugs return "Session not found" with dashboard link.
  4. File watcher extended to .pm/sessions/; no leaked handles on shutdown.
  5. Phase 1 opt-in prompt; yes/no; persists to .pm/config.json.
  6. When enabled, starts dashboard server and opens browser to session URL.
  7. .pm/sessions/ added to .gitignore.
  8. Multiple concurrent sessions supported.
  9. Server binds to 127.0.0.1 only.
PM-061Per-phase companion screens
Users see rich, formatted summaries — scope grids, verdict tables, issue previews — instead of a generic state dump.
Highfeature
Acceptance Criteria (12)
  1. Phase 4: scope grid (in/out columns + 10x badge).
  2. Phase 4.5: reviewer verdict table + blocking issues.
  3. Phase 5: decomposition table + issue cards + Mermaid diagram (client-side rendering).
  4. Phase 5.5: team review verdict table + iteration count.
  5. Phase 5.7: bar raiser verdict + conditions.
  6. Phase 5.8: final proposal link + session summary.
  7. Non-visual phases: placeholder with phase stepper (checkmarks + greyed upcoming).
  8. Shared template defined in skills/groom/references/companion-template.md.
  9. Stepper uses past-tense for completed, present-continuous for current.
  10. Every screen must contain content not available in the terminal.
  11. Conditional on visual_companion: true.
  12. Each phase overwrites current.html.
PM-062Dashboard links + session cleanup
Users discover active companion sessions from the dashboard home; completed sessions are cleaned up automatically.
Mediuminfrastructure
Acceptance Criteria (5)
  1. Dashboard home session banner links to /session/{slug} when companion is active.
  2. No link when companion directory doesn't exist.
  3. Phase 6 deletes .pm/sessions/groom-{slug}/ after issue creation.
  4. Cleanup is silent; no user prompt.
  5. Groom state file is NOT deleted by cleanup.

Advisory Notes

  • Feature is UX polish, not handoff quality — position honestly as a Priority 1 supporting initiative
  • No user evidence for the pain — built on pattern analysis, not demand signal
  • Shareable artifact angle (post-session HTML report) worth logging as a follow-on
  • Codebase annotations in companion view (linking scope items to file paths) — future differentiator

? Open Questions

Questions the decision-maker should be prepared to discuss.

  1. What happens when a user closes the browser tab mid-session and reopens it? (Route reads state file as fallback — but reconnection UX needs clarification.)
  2. How does this interact with the groom-centric dashboard rewrite (PM-056)? (Dashboard home = overview; /session/{slug} = live detail. Clarify hierarchy.)
  3. What is the fallback experience if companion HTML write fails? (Route handler serves rendered state file — but user sees a quality downgrade.)
  4. Why WebSocket over SSE when research recommends SSE? (Pragmatic reuse of existing infrastructure; targeted SSE is a v2 optimization.)

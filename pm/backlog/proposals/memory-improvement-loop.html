! Problem & Context

Every PM groom session generates valuable learnings — what research was useful, which scope decisions got challenged, what review patterns emerged. Today, all of this is discarded when the groom state file is deleted at Phase 6 completion. The next session on the same project starts cold, repeating mistakes and rediscovering context that was already known.

The existing workaround is a manually-maintained learnings.md journal, but it's updated sporadically and never read back into sessions. Claude's built-in session memory captures 5 feedback entries but they're generic, not structured for PM workflows.

Research signal: AI agent memory systems have matured rapidly in 2025-2026. GitHub Copilot's memory system shows 7% improvement in PR merge rates. Trajectory-informed memory extraction shows +14.3pp on task completion. The critical insight across all research: capturing signal is easy; closing the loop back to behavior change is hard. No PM tool has a structured improvement loop.

▢ Scope Overview

  • Project memory file (pm/memory.md) — persistent per-project learnings with defined schema

  • Retro prompt at session end — 3-question structured reflection written to project memory

  • Vector/semantic search (overkill at this scale)
  • Automatic skill prompt modification (too risky for v1)
  • Cloud sync / shared team memory (non-goal per strategy)

☰ Issue Breakdown

PM-038 Project Memory System

The PM plugin remembers what it learned from past grooming sessions and surfaces relevant context at the start of future sessions, so each session starts smarter instead of from scratch.

memory core-loop high priority
  1. pm/memory.md is created and populated when the first groom session completes Phase 6.
  2. Subsequent sessions read pm/memory.md at Phase 1 and surface relevant one-line summaries.
  3. Memory file lives in the project repo, version-controlled alongside the codebase. Memory never leaves the user's repository.
  4. The retro prompt captures qualitative learnings; automated extraction captures quantitative signals.
  5. Memory entries have dates and source citations (which session, which phase).
  6. Phase 6 completion always results in at least one memory entry, or a logged warning if writing fails.
  7. Users can manually add entries following the schema.
PM-039 Define Project Memory File Schema

pm/memory.md has a defined, validated schema so learnings are structured consistently and retrievable by future sessions.

memory schema high priority
  1. YAML frontmatter with type: project-memory, created, updated, and entries: key containing a list of entry objects.
  2. Each entry: date, source, category (scope|research|review|process|quality), learning. Optional: detail.
  3. validate.js exits non-zero with descriptive error when entries are missing required fields.
  4. validate.js prints warning when file exceeds 50 entries.
PM-040 Retro Prompt at Groom Session End

Users reflect on what worked and what didn't at the end of each groom session, and those learnings are written to pm/memory.md automatically.

memory capture high priority
  1. phase-6-link.md modified: retro step inserted after issue creation, before state file deletion.
  2. 3 questions with fixed categories: "What worked well?" (quality), "What was slow?" (process), "What to do differently?" (process).
  3. Skip on any question bypasses all remaining questions gracefully.
  4. State file NOT deleted until retro completes or is skipped.
PM-041 Automated Learning Extraction from Groom State

Structured learnings are automatically extracted from the groom session state file at Phase 6, capturing quantitative signals the retro prompt misses.

memory capture medium priority
  1. Extraction step in phase-6-link.md after retro, before state deletion.
  2. Extracts when: scope iterations >1, team review has conditions, bar raiser sends back, scope items moved.
  3. Clean session marker when scope passed first iteration and bar raiser approved.
  4. Sessions with no notable signals produce no extraction entries.
PM-042 Memory Injection at Phase 1 Session Start

New groom sessions surface relevant learnings from past sessions at Phase 1, so the user starts with accumulated project context instead of from scratch.

memory retrieval high priority
  1. New step in phase-1-intake.md after research check, before codebase scan.
  2. Surface 5 most recent entries by date (deterministic, no relevance ranking in v1).
  3. Memory surfaced before first scope input, not as ambient preamble.
  4. Token budget: ~500 tokens for summaries. Full detail on-demand via fenced blockquote.

? Open Questions

Questions the decision-maker should be prepared to discuss:

  • How do we measure whether memory actually improves outcomes? (v1 captures data; measurement instrumentation is v2. Proxy: at least one memory entry per session with >1 scope iteration.)
  • What is the migration path for the existing learnings.md? (Different purpose: learnings.md captures plugin development retrospectives; pm/memory.md captures product grooming learnings. No migration needed — they coexist.)
  • What happens when memory entries conflict or become stale? (v1 provides manual pruning guideline at 50 entries / 6 months. Automated contradiction detection deferred to v2.)
  • How will we track retro prompt completion rate? (Qualitative observation in v1 — if users consistently skip, we know from the memory file having few retro-sourced entries.)

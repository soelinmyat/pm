! Problem & Context

The product engineer running multiple terminals loses track of what each session is doing.

When groom is running in one terminal, dev in another, and ship in a third, the only way to check status is to alt-tab between windows. There is no central view. Milestone events — a PR created, tests passing, a review completing — are invisible unless the engineer is watching the right terminal at the right moment.

Research signal: OpenCode (SST) validates the SSE event bus architecture at production scale — typed events, POST ingestion, multi-client subscription. No profiled competitor has a real-time multi-terminal activity feed. ChatPRD is browser-only with no terminal awareness. PM Skills Marketplace is stateless per session. Productboard Spark runs isolated Jobs with no cross-session visibility. This is a defensible differentiator.

PM-088 SSE Event Bus + Activity Feed

The product engineer monitors all active terminal sessions from the dashboard without switching windows — key milestones are surfaced proactively so nothing is missed.

infrastructure feature ui high priority
Show acceptance criteria (7)
  1. Terminal sessions can POST events to the dashboard server via POST /events.
  2. The dashboard browser tab receives events in real-time via SSE (GET /events).
  3. The dashboard home page shows a right-sidebar activity feed panel with reverse-chronological events.
  4. Milestone events trigger a toast notification that auto-dismisses after 3–5 seconds.
  5. A port discovery utility lets skills find the running dashboard server port.
  6. pm:ship, pm:dev, and pm:groom emit milestone events during their workflows.
  7. The feed shows an empty state when no events have been received.
PM-090 SSE Event Bus Core — POST, Store, SSE, Port Discovery

Any process on the machine can push an event to the dashboard server and any browser tab can receive it in real-time — the event pipeline works end-to-end.

infrastructure high priority
Show acceptance criteria (10)
  1. POST /events accepts JSON body: type (required), source (required), timestamp (required), detail (optional), source_type (optional, default "terminal").
  2. POST /events returns 201 on success, 400 on missing required fields, 405 if not POST.
  3. Events stored in an in-memory ring buffer capped at 200 events; oldest evicted when full.
  4. GET /events returns Content-Type: text/event-stream with Cache-Control: no-cache and Connection: keep-alive.
  5. Each SSE message uses format id: {id}\ndata: {json}\n\n.
  6. Heartbeat comment (: keepalive\n\n) sent every 15 seconds to prevent proxy/browser timeout.
  7. On reconnect, if client sends Last-Event-ID, server replays missed events from ring buffer.
  8. SSE connections tracked in existing allConnections set and destroyed on server.close().
  9. scripts/find-dashboard-port.sh outputs port on stdout (exit 0) or nothing (exit 1) if no server is running.
  10. Port derived from project directory path using same hashing algorithm as start-server.sh lines 119–123 — no port scanning.
PM-091 Activity Feed Panel on Dashboard Home

The dashboard home page shows a live-updating activity feed in a right sidebar panel — the user sees what every terminal is doing at a glance.

ui feature high priority
Show acceptance criteria (12)
  1. Dashboard home renders a 260px right sidebar panel alongside existing main content.
  2. Panel header shows "Activity" label and a pulsing green "Live" indicator when SSE is connected.
  3. Events render reverse-chronologically with: colored dot (by event type), source label, description, relative timestamp.
  4. Events from last 3 minutes render at full contrast; older events fade to muted text and dimmer dots.
  5. An "Earlier" time separator visually groups older events.
  6. Panel background is slightly darker than main content (#12141a in dark mode) for visual recession.
  7. Browser-side JS creates an EventSource('/events') connection and appends new events to feed DOM in real-time.
  8. On SSE reconnect, feed reconciles state from ring buffer replay via Last-Event-ID.
  9. Empty state: "No events yet. Activity appears here when terminal sessions are running." — Live indicator shows "Idle".
  10. On screens narrower than 1024px, feed panel is hidden.
  11. dashboardPage() extended with conditional sidebar slot; non-Home pages render without it.
  12. On WebSocket-triggered page reload, feed rebuilt from SSE replay via Last-Event-ID.
PM-092 Toast Notifications for Milestone Events

When a terminal session hits a milestone (PR created, tests passed, merged), a brief toast slides in on the dashboard — the user gets a celebration moment without leaving their current view.

ui feature medium priority
Show acceptance criteria (10)
  1. Toast notifications appear in the bottom-right of the main content area (not over the feed panel).
  2. Toasts trigger only for milestone event types: tests_passed, pr_created, review_done, merged.
  3. Each toast shows a status icon and a one-line message (max 10 words).
  4. Toasts auto-dismiss after 500ms per word + 1000ms buffer (minimum 3s, maximum 5s).
  5. Toasts slide in from bottom: translateY 8px → 0, 300ms ease-out.
  6. When prefers-reduced-motion is active, toast appears instantly without animation.
  7. Multiple toasts queue vertically — max 3 visible; oldest dismissed if 4th arrives.
  8. Toasts contain no interactive elements — pure confirmation (WCAG compliant).
  9. Toast styling uses var(--surface), var(--border), var(--shadow-md) — no aggressive colors.
  10. Toast JS listens to the same EventSource connection used by the activity feed (PM-091).
PM-093 Instrument Skills to Emit Milestone Events

When pm:ship, pm:dev, or pm:groom hit key milestones, the dashboard feed shows what happened — the pipeline produces real events, not just plumbing.

feature integration high priority
Show acceptance criteria (9)
  1. pm:ship emits: pr_created, review_done (with score), tests_passed/tests_failed, merged.
  2. pm:dev emits: dev_started (with issue title), tests_passed/tests_failed, phase_started (implement, review, etc.).
  3. pm:groom emits: groom_started (with topic), phase_started (intake, research, scope, etc.), groom_complete (with issue count).
  4. Each event uses the schema from PM-090: type, source, timestamp, detail, source_type: "terminal".
  5. The source field uses the session slug when available, otherwise falls back to "{skill}-{PID}".
  6. Events sent via port discovery utility from PM-090. If no server running, event silently dropped — no error, no retry.
  7. Event emission is non-blocking — skill execution never delayed by a failed POST.
  8. scripts/emit-event.sh wraps port discovery + POST in one call.
  9. Instrumentation added as explicit Bash tool calls in SKILL.md files at specific milestone points.

? Open Questions

Four bar raiser questions for the decision-maker before approving.

  • Events in the ring buffer are lost on server restart — the feed goes empty. Is in-memory only acceptable for v1, or does the server restart limitation block real-world usefulness?
  • Success criteria is "dashboard tab stays open during 50%+ of multi-terminal sessions within 90 days." How will this be measured without analytics instrumentation in scope for this epic?
  • The dashboard now has two real-time channels: WebSocket (page reload) and SSE (event streaming). Is this two-channel design the long-term architecture, or is SSE-only consolidation the intended future state?
  • The activity feed surfaces terminal lifecycle events — is this the path to becoming the messaging differentiator, or does the roadmap call for a separate, more opinionated notification strategy?

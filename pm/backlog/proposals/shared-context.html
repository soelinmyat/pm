1 Problem & Context

PM's knowledge base is local-only, limiting it to a solo tool and blocking monetization.

Research, strategy, competitive intel, and groomed issues live in each user's local pm/ folder. Teammates can't benefit from each other's work. This blocks async product collaboration and limits PM's addressable market to individual product engineers.

Research signal: No AI coding tool shares product knowledge today. Cursor, Windsurf, Copilot, and Claude Code all share coding context (rules files in git). None shares research, strategy, or groomed issues across a team. 26 sources across 5 research areas confirm the gap.

The strategy (Section 5) explicitly positions this as the expansion path: "Free plugin is the acquisition engine; the cloud product is the monetization layer."

2 Scope

Phased v0 (single user remote) + v1 (team sharing), addressing scope review feedback that the original scope was too large.

In Scope

  • v0: Storage abstraction for server.js
  • v0: API server + GitHub OAuth device flow
  • v0: S3 storage (versioned bucket)
  • v0: Postgres metadata (3 tables)
  • v0: CLI: pm login, pm push, pm pull, pm status
  • v0: Project auto-detection from git remote
  • v0: Project bootstrap + deployment
  • v1: Team sharing + invite flow
  • v1: Hosted dashboard (read-only)
  • v1: Billing (Stripe)
  • v1: Changelog + pm restore
10x — No competitor has this

Out of Scope

  • Auto-sync (start with explicit push/pull)
  • Access controls (all-or-nothing for v1)
  • E2E encryption
  • Self-hosted option
  • Real-time collaboration (CRDTs)
  • Enterprise tier (SSO/SAML)
  • Dashboard writes (read-only only)
  • Cross-project shared strategy
  • Email/Gmail auth (GitHub only)

5 Technical Feasibility

Feasible with caveats. The storage abstraction is the critical path.

Build-on

  • server.js dashboard (4,134 lines, full rendering pipeline)
  • createDashboardServer(pmDir) factory pattern
  • parseFrontmatter() YAML parser
  • pm/ file structure (each concern in own file)
  • analytics-log.sh append-only JSONL pattern
  • node:test infrastructure

Build-new

  • StorageProvider interface + S3 implementation
  • API server with auth middleware
  • GitHub OAuth (device flow + web flow)
  • Postgres schema + migrations
  • CLI commands (login, push, pull, status)
  • Stripe billing integration
  • Deployment infra (Dockerfile, CI/CD)

Risks

  • 147 sync fs calls to convert to async (not 80+)
  • Zero-dependency posture breaks (S3 SDK, pg, jose)
  • server.js monolith (4,134 lines) needs splitting
  • Agent-merge logic is novel — no precedent
  • Two OAuth flows needed (device + web)

Sequencing

  1. Project bootstrap (package.json, docker-compose)
  2. Storage abstraction (foundational)
  3. API server + auth
  4. S3 backend + Postgres metadata (parallel)
  5. CLI commands
  6. Project auto-detection
  7. Deployment
  8. v1: Team, dashboard, billing, changelog

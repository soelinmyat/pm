Problem & Context

Phase 5 (Groom) is the weakest link in a pipeline where the review phases are already strong. The current issue drafting step is a template checklist — it tells the drafter what fields to fill in but provides no guidance on how to think about decomposition, quality, or slicing.

The irony: Phase 5.5 reviewers have detailed BAD/GOOD examples for outcomes, ACs, and decomposition — but the drafter in Phase 5 has none. The person writing the issues has less guidance than the person critiquing them.

This means structural defects (horizontal slicing, missing dependencies, vague ACs, task-descriptions-as-outcomes) are systematically produced in Phase 5 and caught in Phase 5.5 — wasting review iterations on problems that should never have been drafted.

Research sources: 9 established story splitting patterns (Humanizing Work), INVEST criteria (Agile Alliance), vertical slicing consensus (Scrum.org), superpowers brainstorming skill (2-3 approach proposals), dev-epic plugin (layer-aware decomposition).

Scope Overview

In Scope

  • Decomposition step (new Step 3) — explore 2-3 approaches, apply splitting patterns, boundary quality test
  • Story splitting patterns reference file — 9 patterns + vertical slicing guidance
  • INVEST validation with evidence trails — forcing function during decomposition, not post-hoc
  • Good/bad examples for the drafter — match Phase 5.5 reviewer quality
  • MVP slicing guidance — thinnest vertical slice per issue
  • Dependency mapping — conditional (4+ issues), with sequencing rationale
10x Differentiator

Out of Scope

  • Size-based ceremony scaling (XS-XL) — changes overall pipeline
  • Confidence tiers on review findings — improves review phases, not drafting
  • Changes to review agent prompts (5.5, 5.7) — already strong
  • Visual companion for decomposition — separate initiative
  • New feature types — current 5 types are adequate

Issue Breakdown

PM-032 Add Decomposition Methodology with Splitting Patterns
PMs receive an issue set with visible decomposition reasoning — the drafter shows which splitting patterns it considered and why it chose one — so they can catch structural problems before engineering picks up the work.
High grooming-quality infrastructure
  1. New reference file at skills/groom/references/splitting-patterns.md encoding 9 Humanizing Work patterns + vertical slicing + meta-pattern
  2. New Step 3 ("Decompose") with explicit Read instruction, 2-3 candidate approaches evaluated against accumulated context
  3. User sees eliminated approaches with rejection reasons — visible interaction, primary differentiator
  4. Boundary quality test: "Can each issue be understood alone? Can one change without breaking another?"
  5. MVP slicing woven in: "thinnest vertical slice delivering end-to-end value" per issue
  6. Good/bad decomposition examples (horizontal slice BAD, no-split BAD, vertical slices GOOD)
  7. Decomposition rationale cites prior phases with fallback for absent phases
  8. Good/bad examples added for outcome statements and ACs in drafting step
  9. Consistent step renumbering with Read instruction in single implementation
  10. New references/ directory (distinct from templates/)
  11. Total new content ≤150 lines; reference file loaded on demand
PM-033 Add INVEST Validation Gate and Dependency Mapping
PMs can trust that every drafted issue is independently scoped, valuable, and testable — with visible evidence for each INVEST dimension — so they spend review time on strategic judgment rather than catching structural defects.
High grooming-quality infrastructure
  1. INVEST validation step inserted between "Decompose" and "Draft issues" (by name, not number)
  2. Per-dimension evidence trails: Independent, Negotiable, Valuable, Estimable, Small, Testable — each grounded in prior-phase context
  3. "Independent" handles dependency tension: "can be understood independently, not has zero dependencies"
  4. Conditional dependency mapping (4+ issues): dependency list, sequencing rationale, parallelization
  5. Skip condition: "3 or fewer issues — sequencing is implicit"
  6. Dependency output feeds Technical Feasibility section of each issue
  7. Bare pass/fail without evidence citation is explicitly rejected; fallback for absent phases
  8. Total new content ≤50 lines in phase-5-groom.md

Advisory Items

  • Monitor for superficial "show your work" reasoning — LLM theater risk if decomposition rationales become boilerplate
  • Splitting patterns reference file needs an ownership/update plan for long-term maintenance
  • Consider tightening ChatPRD comparison — it's Adjacent, not Direct; PM Skills Marketplace is the sharper foil
  • Success metric baseline (Phase 5.5 iteration count) should be measured across 3-5 sessions before shipping

Open Questions

Questions a decision-maker should be prepared to discuss before approving:

  • How will we detect low-quality decomposition reasoning versus genuine analysis? What does "LLM theater" look like in practice, and what is the intervention?
  • Who owns the splitting patterns reference file long-term? Is it a snapshot or a living document? What triggers an update?
  • The success metric (Phase 5.5 iteration reduction) requires a pre-ship baseline. Who observes the next 3-5 sessions to establish it, and does that delay shipping?

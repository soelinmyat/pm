M Draft 3 Issues

Skill Step Flow Guidance

Add consistent step-transition flow guidance across all 15 skills so users never need to manually prompt "continue."

Findings Architecture Decisions Risks Issues Change Log

Codebase Findings

references/skill-runtime.md

Defines step loading and execution order but has no section on step transitions. The "Interaction Pacing" section covers question cadence but not step-to-step advancement.

AGENTS.md — Step Authoring Rules

Requires every step to have Goal, How, and Done-when sections. Done-when defines exit criteria but does not require forward guidance. This is the gap — Done-when tells the agent when a step is complete but not what to do next.

skills/*/steps/*.md — 65 step files across 15 skills

Analysis of all 65 step files reveals three patterns:

  • Good (6 skills): groom, rfc, dev, think, ideate, strategy — final steps offer explicit next actions with exact commands
  • Weak (3 skills): ingest, start, ship — offer vague recommendations without branching
  • Missing (6 skills): note, refresh, research, setup, sync, using-pm — final steps end with reports or confirmations, no forward guidance
Good pattern: groom step 11, rfc step 3, think step 6

These follow a consistent template: summarize what was done, name the artifact path, offer the exact next command. Example from groom: "Grooming complete for '{topic}'. Next: run pm:rfc {slug} to generate the RFC."

Mid-step transitions

Steps with HARD-GATEs (groom research, groom scope, rfc intake) never stall — the gate blocks advancement explicitly. Steps without gates (most mid-steps) rely on the agent to notice the Done-when conditions are met and advance. This is where "continue" prompting happens — the agent completes work but doesn't know to advance.

Architecture

This is a convention change, not an infrastructure change. Three layers of modification:

Layer 1: Convention (skill-runtime.md)

Add a new "Step Transitions" section to references/skill-runtime.md that codifies two patterns:

  • Mid-step advancement: Every Done-when section must end with an explicit "Advance to Step N" directive
  • Final-step completion: Every skill's last step must summarize what was done and offer the next action

Layer 2: Authoring Rules (AGENTS.md)

Update the Step Authoring Rules section in AGENTS.md to add a fourth required element alongside Goal/How/Done-when:

  • Advance: What happens after Done-when is satisfied. For mid-steps: "Advance to Step N ({name})." For final steps: summary + next action.

Layer 3: Step Files (skills/*/steps/*.md)

Update step files that lack explicit advancement language. This is the bulk of the work. The existing good patterns (groom, rfc, think) are the template — not inventing something new.

Apps/services affected: references/skill-runtime.md, AGENTS.md, 65 step files across 15 skills

Cross-boundary sync required: No

Key Decisions

Convention over engine change

Chosen: Add advancement language to step file content (markdown text the agent reads).

Rejected: Add a step-loader mechanism that auto-injects "advance to next step" between steps.

Rationale: Step files are markdown instructions read by the agent. The agent doesn't have a "step engine" — it reads each file and follows instructions. Adding advancement language to the content is the only mechanism available. An engine change would require modifying the Claude Code plugin framework itself, which is out of scope.

Append to Done-when vs. separate Advance section

Chosen: Append advancement directive to the end of each Done-when section rather than adding a separate "Advance" section.

Rejected: A new ## Advance section after Done-when.

Rationale: The agent reads Done-when to know when to stop. Placing the advancement directive there ensures it reads "you're done, now go here" in one place. A separate section risks the agent stopping at Done-when and never reaching the Advance section — the exact problem we're fixing.

Final-step template: summary + choice

Chosen: Final steps show a brief summary of what was accomplished, then offer the logical next action (or ask "what's next?" if no clear single path).

Rejected: Always offering a menu of options. Also rejected: never asking, always auto-routing.

Rationale: Some skills have one clear next step (groom -> rfc -> dev). Others have branching paths (ideate -> groom one or explore more). The template adapts: single next step uses "Next: run X", branching uses "Want to Y?".

Risks

RiskImpactMitigation
Advancement directives make steps too prescriptive, reducing agent flexibility Low Directives say "advance to Step N" not "you must do X" — they guide flow, not content
Large number of files to edit (65 step files) increases risk of inconsistency Medium Issue 1 establishes the pattern template first. Issues 2-3 apply it systematically. Review catches drift.
Some step files have complex conditional advancement (e.g., tier-dependent skips) Low Steps with existing HARD-GATEs already handle this. Only add advancement to steps that currently lack it.

Issues

1 Codify step-transition convention S

Outcome

The step-transition convention is documented in skill-runtime.md and AGENTS.md, giving all future step files a clear template for advancement language.

Acceptance Criteria

  1. references/skill-runtime.md has a new "Step Transitions" section with mid-step and final-step patterns
  2. AGENTS.md Step Authoring Rules updated to require advancement language in Done-when
  3. Both files committed on the feature branch
  4. Existing tests still pass

Approach

File: references/skill-runtime.md — Add a "Step Transitions" section after "Interaction Pacing" with two subsections:

## Step Transitions

### Mid-step advancement

Every Done-when section must end with an explicit advancement directive:

> Advance to Step {N} ({step-name}).

If the step has conditional paths (tier-dependent skips, user choices),
each path must name its destination:

> If quick tier: advance to Step 4 (Scope). If standard/full: advance to Step 3 (Research).

### Final-step completion

The last step of every skill must:

1. **Summarize** what was accomplished (1-2 sentences naming the artifact)
2. **Offer the next action:**
   - Single clear path: "Next: run `pm:{skill} {slug}` to {what it does}."
   - Branching paths: Ask one question with named options.
   - Everything done: "Complete. What would you like to do next?"

File: AGENTS.md — In "Step Authoring Rules > Required sections", add advancement language to Done-when description:

3. **Done-when** — exit criteria so the agent knows when to advance.
   Must end with an explicit advancement directive:
   mid-steps say "Advance to Step N ({name})";
   final steps summarize + offer next action.
   Without this, the agent either lingers or waits for manual prompting.

3 Add mid-step advancement directives S

Outcome

Every non-final step across all 15 skills ends with an explicit "Advance to Step N" directive. No step leaves the agent wondering what comes next.

Acceptance Criteria

  1. Every non-final step's Done-when section ends with "Advance to Step N ({name})" or conditional equivalent
  2. Steps with existing HARD-GATEs or explicit advancement are left unchanged
  3. Steps with tier-dependent skips include conditional advancement for each path
  4. Existing tests still pass

Approach

Systematically scan every non-final step file. For each, check if Done-when already includes advancement language. If not, append the appropriate directive.

Pattern for simple advancement:

## Done-when
- {existing criteria}

Advance to Step {N} ({step-name}).

Pattern for conditional advancement:

## Done-when
- {existing criteria}

If {condition A}: advance to Step {N}. If {condition B}: advance to Step {M}.

Skip list (already have explicit advancement or HARD-GATEs):

  • groom steps 1-10: most have explicit gates or advancement
  • rfc step 1: has explicit routing to step 2
  • dev steps 1-4: have explicit stage routing
  • think steps 1-5: have sequential advancement

Priority targets (steps known to cause stalling):

  • refresh steps 1-4: audit/execute steps with no forward guidance
  • research steps 1-4: mode routing and topic steps with weak advancement
  • sync steps 1-4: implementation steps with no gates
  • note step 1: capture step offers enrichment but doesn't advance
  • setup steps 1-2: parsing/config steps with no advancement

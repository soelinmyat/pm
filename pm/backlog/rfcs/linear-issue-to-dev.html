Codebase Findings

Five findings from exploring the three skill files. These shaped the approach and issue boundaries.

skills/dev/SKILL.md — Route Detection, Step 2
Local backlog resolution runs first. Step 2 checks pm/backlog/{slug}.md, then scans frontmatter for matching id: fields. Only after both local checks fail does it fall through to "Epic detection (MCP fallback)" which fetches via MCP and checks for sub-issues. Single issues from Linear that have no local backlog match currently get routed with the raw argument as "topic" — no readiness check, no structured context. This is the insertion point for the new Linear fetch-and-assess logic.
skills/dev/references/single-issue-flow.md — Stage 1, Step 3 + Stage 2.5
Intake already fetches tickets but never assesses readiness. Stage 1 Step 3 says: "Issue tracker ticket ID provided? Fetch via MCP." But there is no quality gate between fetching and proceeding. Stage 2.5 (RFC Check) assumes a local pm/backlog/{slug}.md exists — its Step 1 reads frontmatter for handoff_ready and rfc fields. Linear-sourced issues that bypass groom will not have this file. Stage 2.5 needs a parallel path for dev-ready Linear issues that have no local backlog entry.
skills/groom/phases/phase-1-intake.md
Phase 1 handles "existing idea from backlog" but not "existing Linear issue." The existing backlog path reads pm/backlog/{slug}.md with status: idea and pre-fills intake. A similar path for Linear issues would read the issue title + description from MCP and pre-fill, skipping clarification questions when the description is rich enough. This is the Linear-seeded groom entry point.
skills/groom/phases/phase-6-link.md — Step 2
Phase 6 creates Linear issues but never updates existing ones. When Linear is configured, Phase 6 creates a new parent issue. It does not handle the case where the Linear issue already exists (because groom was triggered by dev's readiness check on an existing Linear issue). Needs a write-back path: comment with groom output + update description with enriched AC.
skills/ship/SKILL.md + skills/dev/references/single-issue-flow.md — Knowledge Base Updates
Post-merge knowledge base updates assume a local backlog entry already exists. The section says: "If pm/backlog/{slug}.md exists, update its frontmatter." For Linear-originated work with no local backlog, nothing happens — product memory is lost. Ship needs a creation path (not just update) for Linear-sourced work, with linear_id in frontmatter.
ℹ
Linear MCP tools already available: get_issue (fetch), save_issue (update description/fields), save_comment (write comment), list_issue_statuses (get status options). No new MCP capabilities needed.

Why modify existing files instead of creating a new reference file

The Linear integration touches three points in the existing workflow (intake, groom, ship). Creating a separate linear-integration.md reference would force every flow to check "am I a Linear issue?" and load an extra file. Instead, the Linear path is woven into each skill's existing decision tree — one more branch in the routing logic. This is how the epic detection already works: inline in the routing, not a separate file.

Why linear_id lives in the session state, not a stub file

The proposal says "no local file at intake." The linear_id needs to survive from intake to ship. The dev session state file (.pm/dev-sessions/{slug}.md) already persists across the full dev lifecycle and is read at every stage transition. Adding linear_id there is zero overhead. Ship reads the session state to know what to write back.

Key Decisions

Three design decisions. Each shaped by codebase findings and the proposal's principles.

Generous dev-readiness check Chosen
Accept testable statements anywhere in the description, not just under "AC:" headers. Check for: (1) acceptance criteria exist as testable statements, (2) scope is inferable from the description, (3) size can be classified. All three must pass.
Pros
  • Works with varied Linear description formats
  • Well-groomed issues skip ceremony immediately
  • No need to enforce a description template in Linear
Cons
  • May let through marginally-ready issues
  • Judgment call by the LLM, not deterministic
Strict AC header matching Rejected
Require "Acceptance Criteria" or "AC:" header in the Linear description. Parse only structured content under that header.
Pros
  • Deterministic, no ambiguity
  • Enforces consistent Linear hygiene
Cons
  • Most Linear issues use free-form descriptions
  • Would cause nearly all issues to fail readiness, making the fast-path useless
Groom writes both comment and description to Linear Chosen
When auto-groom enriches a thin Linear issue: (1) write the full groom output as a comment (preserves original description), (2) append enriched AC to the issue description. Team sees the enrichment where they work.
Pros
  • Comment preserves history and audit trail
  • Updated description makes issue self-contained
  • Both channels inform the team
Cons
  • Two MCP calls instead of one
  • Description edit could surprise authors
Product memory created at ship, not intake Chosen
The pm/backlog/{slug}.md file for Linear-originated work is only created post-merge by the ship skill. No stub at intake. The RFC is the local artifact during development.
Pros
  • Knowledge base records outcomes, not intentions
  • Abandoned work leaves no trace (cleaner pm/)
  • Matches proposal principle: "pm/ records outcomes"
Cons
  • Dashboard does not show in-progress Linear work
  • If ship is skipped, no record exists

Technical Risks

Risk Impact Mitigation
Linear MCP unavailable at intake Dev cannot fetch issue data. No fallback context exists because no local file was created. Catch MCP error, ask user to paste issue description manually. Log linear_fetch: failed in session state and proceed as a conversation-sourced task.
Most issues fail readiness The fast-path (skip groom) is rarely used. Every Linear issue triggers auto-groom, adding 5-15 minutes. Accepted trade-off. Thin issues need grooming regardless. The readiness check prevents wasted RFC generation on vague requirements.
Groom write-back overwrites manual edits A teammate edits the Linear description after groom fetched it. Groom's save_issue overwrites the teammate's change. Append enriched AC below a separator (---\n## Enriched AC (auto-groom)) instead of replacing the description. Original content is preserved.
Session state loses linear_id If the session file is corrupted or deleted, ship does not know this was a Linear-originated issue. No product memory written. Also store linear_id in the RFC metadata (<script type="application/json"> block). Ship checks both sources.

Dev intake: Linear fetch, readiness assessment, and routing

Size: M · Depends on: none

Outcome: When a user says "build PM-45" and no local backlog entry exists, dev fetches the Linear issue, assesses whether it has AC + scope + size, and either routes to RFC planning (ready) or triggers auto-groom (not ready). The Linear issue ID is stored in the session state for downstream use by groom and ship.

Acceptance Criteria
  1. When $ARGUMENTS matches an issue ID pattern (e.g., PM-45, CLE-200) and no local pm/backlog/{slug}.md or frontmatter id: match is found, dev fetches the issue via Linear MCP get_issue.
  2. After fetch, dev assesses dev-readiness by checking: (a) testable acceptance criteria exist in the description, (b) scope is inferable, (c) size can be classified. All three must pass.
  3. If all three pass, dev sets linear_id, linear_title, linear_description, and linear_labels in .pm/dev-sessions/{slug}.md and proceeds to Stage 2.5 — the RFC check recognizes this as a Linear-sourced dev-ready issue and skips to RFC generation.
  4. If any check fails, dev also classifies size (XS/S/M/L/XL) from the available context. For XS/S: handle inline — confirm scope + ACs with the user conversationally (same as existing XS/S ungroomed path), storing linear_id in session state. For M/L/XL: announce the gaps and invoke pm:groom within the same conversation, passing Linear context (title, description, labels, ID) as conversation context — not CLI flags. Dev specifies the slug for groom to use: “Use slug: {slug}”.
  5. If MCP get_issue fails (timeout, auth error, Linear down), dev catches the error, logs linear_fetch: failed, and asks the user to paste the issue description manually. Session proceeds as conversation-sourced.
  6. The epic detection path (issue with sub-issues) is unchanged — sub-issue check still runs first.
  7. Validation passes: node scripts/validate.js --dir pm succeeds after all file changes.

Resolved Questions

1
How does groom receive Linear context from dev?

A: Conversation context, not CLI flags. Dev passes Linear data (title, description, labels, ID, slug) as conversation text before invoking groom. Groom reads it from preceding messages. This follows the existing inter-skill pattern.

2
What happens for XS/S Linear issues that fail readiness?

A: Handle inline, same as existing XS/S ungroomed path. Confirm scope + ACs conversationally. Only M/L/XL trigger auto-groom.

3
Should groom write-back be idempotent?

A: Yes. If the description already contains “## Enriched AC (auto-groom)”, replace that section instead of appending again.

4
Does the ship backlog template pass validate.js?

A: Fixed. Added priority: medium to the template. This matches the required fields in validate.js.

5
What about slug mismatch between dev and groom?

A: Dev specifies the slug for groom to use via conversation context. Groom uses that slug instead of deriving its own. This ensures both skills write to the same backlog path.

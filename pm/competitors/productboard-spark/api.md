# Productboard -- API

Note: This documents the Productboard platform API. Productboard Spark (the AI agent) does not have its own public API. Spark consumes data via MCP connectors and knowledge base imports, but does not expose an API for external integration.

## API Availability

Public. Productboard offers a RESTful API at `https://api.productboard.com` with two versions:
- **API v1** (stable, production-ready)
- **API v2** (beta, not recommended for production)

Developer portal: `developer.productboard.com`

Note: API access may be restricted by plan tier. Multiple sources report that the public API is only available on higher-tier plans, which has been cited as a lock-in mechanism by users switching away.

## Auth Model

Multiple authentication methods supported:

| Auth Type | Version | Details |
|---|---|---|
| API Key (Bearer Token) | v1, v2 | Personal Access Token generated by workspace admins from Integrations page. Sent via `Authorization: Bearer <token>` header. |
| OAuth 2.0 Authorization Code | v2 | Full OAuth2 flow for user-delegated access. Application registration at `app.productboard.com/oauth2/applications`. |
| OAuth 2.0 JWT Bearer | v2 | Service-to-service authentication using JWT. |

The availability of OAuth 2.0 in API v2 signals maturing integration posture. API v1 relies on simpler API key auth, which is adequate for internal integrations but less suitable for third-party app ecosystems.

## Core Entity Model

| Entity | Description | CRUD Coverage (v1) | CRUD Coverage (v2) |
|---|---|---|---|
| Note | Customer feedback, ideas, insights. Types: simple, conversation, opportunity | Create | Full CRUD + Search |
| Feature | Product feature in the hierarchy | Full CRUD + push to delivery tools | Full CRUD (as Entity type) |
| Component | Product component grouping features | Read | Full CRUD (as Entity type) |
| Product | Top-level product entity | Read | Full CRUD (as Entity type) |
| Subfeature | Child feature under a Feature | Read | Full CRUD (as Entity type) |
| Initiative | Strategic initiative | CRUD | Full CRUD (as Entity type) |
| Objective | Strategic objective (OKR) | CRUD | Full CRUD (as Entity type) |
| Key Result | Measurable result under an objective | CRUD | Full CRUD (as Entity type) |
| Release | Planned release | CRUD | Full CRUD (as Entity type) |
| Release Group | Grouping of releases | CRUD | Full CRUD (as Entity type) |
| Company | Customer company | Full CRUD | -- |
| User | Individual user/contact | Full CRUD | -- |
| Custom Field | Custom data fields | CRUD | -- |
| Member | Workspace member | -- | Read (List/Get) |
| Team | Workspace team | -- | Full CRUD |

API v2 unifies product hierarchy entities (feature, component, product, subfeature, initiative, objective, keyResult, release, releaseGroup) under a single "Entities" resource with type filtering, which is a notable architectural simplification.

### Configuration Endpoints (v2 only)

API v2 introduces `/configurations` endpoints for each resource type, enabling dynamic schema discovery. Integrations can query available fields, metadata, and field usage rules specific to a workspace. This is a mature API pattern that supports workspace customization without breaking integrations.

## Webhooks

- **Availability:** Supported in both v1 and v2.
- **Supported event types:** Feature updates, note creation/updates, objective changes, initiative changes, release updates (v1). V2 offers subscription-based event notifications with broader coverage.
- **Delivery mechanism:** HTTP POST to registered URL.
- **Security:** Probe request confirmation required for webhook registration. No HMAC signature verification documented.
- **Retry behavior:** Not documented.
- **Management:** Create, list, retrieve, delete subscriptions. No update operation (must delete and recreate).

## Rate Limits

- **Limit:** 50 requests per second per access token.
- **Exceeded response:** HTTP 429 Too Many Requests with `Retry-After` header.
- **Response headers:** Each rate-limited response includes headers showing current limit, remaining requests, and reset timing. Follows Kong rate-limiting header standard.
- **Per-plan differentiation:** Not documented, but described as "generous enough for standard integrations" with support for higher limits upon request.
- **Per-endpoint limits:** Not documented.

50 req/s is a reasonable rate limit for mid-market SaaS. It is sufficient for standard integrations but may constrain bulk data sync scenarios.

## SDKs and Integrations

### Official SDKs

No official SDKs documented in any language. The API is raw REST.

### Community/Third-Party

- **Postman collection:** Official Productboard API Reference available on Postman (`pb-stargate/productboard-public-api`).
- **MCP servers:** Community-built MCP servers exist on GitHub (e.g., `Enreign/productboard-mcp`, `kenjihikmatullah/productboard-mcp`) for integrating Productboard API into agentic workflows.

### Native Integrations

**Customer Feedback:** Slack, Microsoft Teams, Salesforce, Zendesk, Intercom, Gong, Amplitude, Mixpanel, FullStory, Gainsight PX/CS, Screeb, Survicate, InSided, Grain, Cobba&iuml;, Email, Chrome Extension

**Development:** Jira (deep integration), Azure DevOps, GitHub, Trello, Shortcut

**Design:** Figma, Miro, Mural

**Automation:** Zapier

**Open APIs:** Notes API (ingest feedback), Features API (export to dev tools), GDPR API (compliance)

### Marketplace

No public app marketplace. Integrations are curated and listed on the integrations page. Third-party developers build via the public API but there is no directory for discovering community-built integrations.

## Architectural Signals

- **Inference: Two-generation API reveals platform evolution.** API v1 is resource-specific with inconsistent CRUD coverage (Notes are create-only, Features are read-heavy). API v2 unifies entities under a single resource with full CRUD and search, showing a significant architectural maturation. The v2 configuration endpoints for dynamic schema discovery are an enterprise-grade pattern.

- **Inference: Notes as the primary integration ingestion point.** The Notes API was the first public API (and remains the most promoted), indicating that Productboard's primary integration value proposition is ingesting customer feedback from external sources. The data model is optimized for "data in" rather than "data out."

- **Inference: No SDK investment signals API as secondary priority.** The absence of official SDKs in any language, despite 6,000+ customers and $262M in funding, suggests the API is not a primary growth vector. Most integration value is delivered through native connectors (Jira, Slack, Intercom) rather than the developer API.

- **Inference: Spark's MCP approach may eventually supersede the REST API for AI-native integrations.** Spark uses MCP connectors to pull external data, and Productboard has proposed an MCP server for exposing product data to AI development tools. This could represent a future architectural direction where LLM-native protocols (MCP) coexist with or replace traditional REST APIs.

- **Inference: Webhook security gap.** No HMAC signature verification, no retry behavior documentation, and no update operation on webhook subscriptions suggest the webhook system is functional but not enterprise-hardened.

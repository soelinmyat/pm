Codebase Findings

What the EM found after reading the codebase. These findings shaped the architecture and issue breakdown.

src/editor/Editor.tsx:42-180
Tiptap v2 with ProseMirror. Editor is already extension-based. The Collaboration and CollaborationCursor extensions can be added without refactoring the editor core. Current extensions: StarterKit, Table, Image, Placeholder, CodeBlock.
src/lib/api.ts:1-35 + src/server/routes/documents.ts
REST-only API. No WebSocket infrastructure exists. Auth uses JWT in headers. We'll need a separate WebSocket server that validates the same JWT tokens. The auth middleware at src/server/middleware/auth.ts exports a validateToken() function that can be reused directly.
src/server/db/schema.ts
Drizzle ORM with Postgres. Documents table has id, title, content (JSON), owner_id, timestamps. The content column stores ProseMirror JSON. For collab, we'll keep this as the "last rendered" snapshot and add a Yjs binary in S3 as the source of truth.
package.json
Tiptap v2.4.0. Already on a version that supports the collaboration extensions. No upgrade needed. Dependencies to add: yjs, @hocuspocus/server, @tiptap/extension-collaboration, @tiptap/extension-collaboration-cursor, y-indexeddb.
ℹ
Phase 0 spike result: Yjs syncs our custom nodes (table, callout, embed) correctly. 50 synthetic users hit p95 sync latency of 147ms on a single Hocuspocus instance. Memory: ~2.1x raw doc size. No showstoppers.

Architecture

Tiptap client (CRDT) ↔ Hocuspocus (WebSocket) ↔ Postgres + S3.

System Overview
graph LR A[Tiptap Client
Yjs CRDT] -->|WebSocket| B[Hocuspocus
Server] B -->|awareness| A B -->|Yjs binary| D[(S3)] B -->|metadata| C[(Postgres)] E[Auth Service] -->|validateToken| B F[REST API] -->|comments, sharing| C

Hocuspocus runs as a sidecar process alongside the REST API. Same deployment, separate port. It authenticates using the existing validateToken() from our auth middleware.

Why Hocuspocus, not custom WebSocket

Hocuspocus handles the entire Yjs sync protocol, document loading/storing lifecycle, and awareness broadcast. Building this from scratch with y-websocket would take ~2 extra weeks and we'd end up reimplementing what Hocuspocus already does. It's ~2K lines — small enough to fork if the project goes unmaintained.

Persistence strategy

Source of truth: Yjs binary in S3 (versioned bucket). Loaded on first connection, stored on last disconnect + every 60s during active editing.

ProseMirror JSON: Still written to Postgres content column on each store. This keeps the existing read path (API, search, dashboard) working without migration.

Offline: y-indexeddb persists the Y.Doc locally. On reconnect, Yjs sync protocol reconciles automatically.

Comments: REST, not CRDT

Comments use a traditional REST API + Postgres, not the Yjs document. Anchored to text ranges via ProseMirror position mapping.

⚠
Trade-off: Comments can't sync offline. Accepted because: comments are read-heavy, need server-side notifications, and would bloat the Y.Doc.

Key Decisions

Three architecture decisions. Each includes alternatives we considered and why they were rejected.

Yjs CRDT + Hocuspocus Chosen
CRDT engine with managed WebSocket server. Tiptap first-party integration.
Pros
  • Offline-first by design, no custom sync
  • First-party Tiptap integration
  • Battle-tested (Linear, etc.)
Cons
  • ~2x memory overhead (tombstones)
  • Hocuspocus is a smaller project
OT via custom server Rejected
Operational Transformation with a central authority. Google Docs approach.
Pros
  • Lower memory footprint
  • Well-understood algorithm
Cons
  • No offline support without custom merge
  • No Tiptap binding — 6-8 weeks extra
  • Central server = single point of failure
Cloudflare Durable Objects Rejected
Per-document state in Durable Objects with WebSocket hibernation.
Pros
  • Auto-scales per-document
  • Built-in hibernation
Cons
  • 128MB limit per DO
  • Vendor lock-in
  • No Yjs integration — custom bridge needed

Advisory Notes

@adversarial-engineer
WebSocket reconnection should use exponential backoff with jitter to avoid thundering-herd on server restart. The default Hocuspocus retry is fixed-interval.
@tester
CRDT merge tests should include a scenario where two users edit the same paragraph simultaneously from different starting states — this is the most common source of unexpected merge artifacts.
@staff-engineer
Consider adding a document size metric early (PM-093). If Y.Doc state exceeds ~5 MB, GC strategy in PM-100 becomes critical-path rather than nice-to-have.

# RFC Template

Reference template for engineering RFCs generated by `pm:dev`. The RFC is the single engineering artifact — it contains the technical approach, issue breakdown, test strategy, and risks. Issues are sections within the RFC, not separate files.

## Frontmatter Schema

```yaml
---
type: rfc
parent: "{proposal-slug}"
status: draft | reviewed | approved
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

## Document Structure

```markdown
---
type: rfc
parent: "{proposal-slug}"
status: draft
created: YYYY-MM-DD
updated: YYYY-MM-DD
---

# RFC: {Feature Title}

> **For agentic workers:** Use dev:subagent-dev to implement issues from this RFC. Each issue maps to a development unit with its own branch, TDD cycle, and PR.

**Proposal:** `pm/backlog/{proposal-slug}.md`
**PRD:** `pm/backlog/proposals/{proposal-slug}.html`

## Context

{1-2 paragraphs: what this feature does, why it matters, key constraints from the proposal.
Reference the PRD for full product context.}

## Approach

{Technical approach at the system level. Architecture decisions, patterns chosen,
libraries selected, data model changes. 2-5 paragraphs.}

**Apps/services affected:** {list affected apps, packages, or services}
**Cross-boundary sync required:** {yes/no}

## Issues

Each issue is an independently implementable unit. Issues are ordered by dependency —
implement top-to-bottom. Each issue should produce working, testable software on its own.

### Issue 1: {Title}

**Outcome:** {What changes when this ships — one sentence.}

**Acceptance Criteria:**
1. {Specific, testable condition}
2. {Specific, testable condition}
3. {Edge cases handled: ...}

**Approach:**
{Which files to create/modify, key implementation details, patterns to follow.
Detailed enough that a developer with zero codebase context can execute.}

**Dependencies:** None | Issue {N}

**Size estimate:** XS | S | M

---

### Issue 2: {Title}

{Same structure as Issue 1}

---

{Repeat for each issue}

## Test Strategy

{Overall testing approach for this feature:
- Unit test boundaries
- Integration test scope
- E2E/smoke tests if applicable
- Test data requirements}

## Risks

{Known risks, mitigations, and open questions:
- Technical risks (performance, compatibility, migration)
- Product risks (edge cases, user confusion)
- Dependencies on external systems or teams}

## Decisions

{Non-obvious technical choices made during RFC creation.
For each: what was decided, what alternatives were considered, why this was chosen.
These may be extracted as ADRs during RFC review.}
```

## Usage Notes

- Dev writes the RFC to `pm/backlog/rfcs/{slug}.md` where `{slug}` matches the proposal slug.
- After RFC is written, dev updates the proposal's frontmatter: `rfc: rfcs/{slug}.md`.
- RFC review (3 reviewers) happens before implementation begins.
- During implementation, the dev state file (`.pm/dev-sessions/{slug}.md`) tracks per-issue progress. The RFC itself is not updated with status — it's the engineering plan, not a tracker.
- Issue sizes within the RFC inform the implementation approach (TDD depth, review gates) per the dev skill's size routing.

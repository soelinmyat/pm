Deep Dive

The standards landscape is converging — but hasn't converged yet

Token Format Standards

The W3C Design Tokens Community Group (DTCG) has published a draft specification for a standard JSON format. It defines token types (color, dimension, font family, etc.), aliasing, and composite tokens. Major tools are adopting it:

  • Figma Variables — export aligns with DTCG structure but uses Figma-specific JSON schema [1]
  • Tokens Studio — migrating from legacy format to DTCG-compatible output [2]
  • Style Dictionary v4 — added DTCG parser as a first-class input format [3]
  • Supernova — imports and exports DTCG format natively [4]
"The DTCG spec isn't finished, but the direction is clear enough to build on. The core token types and aliasing model won't change — only edge cases around composite tokens and conditional values are still being debated." — Jina Anne, co-chair of DTCG, Design Tokens Summit 2025 [5]

However, the spec is still "Editor's Draft" — not a W3C Recommendation. This means tools are adopting the spirit but diverging on details. Our ingestion layer needs to handle DTCG-format, Figma Variables API output, Tokens Studio legacy format, and Style Dictionary JSON.

Token pipelines

Based on the 2025 Design Systems Survey [6] and Sparkbox's Design Systems Report [7], we see a clear maturity ladder:

  • Level 1 — Primitive tokens (73%): Color palette, spacing scale, type scale. These exist as Figma styles or Variables. Not formalized as a pipeline.
  • Level 2 — Semantic tokens (52%): Tokens with meaning (e.g., color-bg-primary vs. blue-500). Usually stored in Figma Variables. Light/dark theme support.
  • Level 3 — Component tokens (31%): Tokens per component (e.g., button-padding-md, card-border-radius). This is where our design-system-aware code gen gets powerful.
  • Level 4 — Token pipeline (18%): Tokens flow from design tool → JSON → code automatically via Style Dictionary or custom pipeline. Most mature teams.

Implication for Flux: Our ICP is Level 2-3 teams. They have tokens in Figma but may not have a pipeline. We should read directly from Figma Variables API (Level 2) and optionally ingest Style Dictionary JSON for Level 4 teams.

Deep Dive

The token tooling ecosystem is mature enough to build on

Tooling Landscape

Three tools matter for our ingestion strategy:

  • Figma Variables API [8] — REST API and Plugin API both expose variables. REST requires Figma Enterprise or Pro plan. Plugin API works for all users. We should support both paths.
  • Tokens Studio [9] — Figma plugin used by ~200K designers to manage tokens. Exports JSON. Many teams in our ICP use it. Supporting Tokens Studio export format catches teams who aren't on Figma Variables yet.
  • Style Dictionary v4 [10] — Token transformation pipeline. The "webpack of design tokens." Level 4 teams use it to transform tokens into CSS variables, Swift constants, Android resources, etc. If a team uses Style Dictionary, they have a tokens.json we can ingest directly.
"Figma Variables are the source of truth for ~60% of teams with design systems. The rest use Tokens Studio or a custom JSON pipeline. If you can read Figma Variables and Tokens Studio JSON, you cover 85%+ of the market." — Nathan Curtis, EightShapes, Design Systems Conference 2025 [11]

Recommended ingestion priority: (1) Figma Variables API, (2) Tokens Studio JSON export, (3) Style Dictionary JSON, (4) DTCG standard format. In that order — maximum coverage for minimum implementation.

So What

What this means for Flux

Build
Ingest Figma Variables API first, Tokens Studio JSON second
These two paths cover 85%+ of our ICP. Figma Variables is zero-friction (no export step). Tokens Studio catches teams not yet on Variables. Style Dictionary and DTCG are stretch goals.
Supports: Bet #1, PM-112 · Based on: Findings 3, 4 + Sources 1, 8, 9, 11
Build
Support multi-theme (light/dark) from day one
Every serious design system has themes. If our token ingestion can't handle modes/themes, we'll be blocked on 80%+ of production design systems. This is not a V2 feature.
Supports: PM-112 · Based on: Finding 5 + Sources 6, 8
Build
Target Level 2-3 teams, not Level 4
Level 4 teams (with pipelines) already have a working system. Level 2-3 teams have tokens but no pipeline — we ARE their pipeline. This is the biggest value-add segment.
Supports: ICP definition · Based on: Finding 2 + Sources 6, 7, 13
Avoid
Don't build a token management tool
Tokens Studio and Figma Variables already handle token creation and management. We ingest tokens, not manage them. Building management UX is a distraction from code generation.
Based on: competitive analysis + Sources 2, 9
Watch
Track DTCG spec finalization timeline
When the spec moves from Editor's Draft to Candidate Recommendation, expect rapid tool convergence. At that point, a single DTCG parser may replace our multi-format ingestion. Don't over-invest in legacy formats.
Based on: Finding 1 + Sources 5, 12
Still Unknown

What we still need to learn

1
How many of our ICP teams use Figma Enterprise (required for REST API) vs. Pro (plugin API only)?
Determines whether we need a plugin-first or API-first ingestion path
2
What percentage of component tokens are defined at the Figma level vs. only in code (CSS variables)?
If component tokens only exist in code, we may need a CSS parser alongside Figma ingestion
3
When Figma ships their own code generation (Config 2026?), will it use Variables natively?
If Figma's code gen uses tokens, our "design system aware" differentiation erodes

Multi-Language Support

Users can switch the interface language and manage translated content, starting with English, Spanish, and Japanese.

Size: M · Created Apr 10, 2026
Outcome Problem Scope User Flows Wireframes Competitive Feasibility Review Resolved Metrics Next Steps

Outcome

Users can switch the app interface between English, Spanish, and Japanese — expanding reach to 3 new markets without creating separate product builds.

Non-English-speaking users currently bounce at onboarding because every label, tooltip, and help string is hardcoded in English. After this ships, the language picker in Settings lets users switch instantly. Content they create stays in whatever language they write it in; only the chrome translates.

Problem & Context

28% of trial signups come from non-English markets, but only 4% convert — a 7x gap vs. English users.

"Three enterprise prospects in Japan asked for Japanese UI before signing POC agreements." — Sales call notes, March 2026

The strategic driver is geographic expansion (Q2 priority). Competitors Acme and Globex already ship 5+ languages. We're losing deals to localized alternatives in LATAM and APAC.

Scope

In Scope

  • i18n framework (ICU MessageFormat)
  • Language picker in Settings
  • Extract all UI strings to locale files
  • English, Spanish, Japanese translations
  • RTL-aware layout foundation

Out of Scope

  • User-generated content translation
  • RTL languages (Arabic, Hebrew) — layout prep only
  • Auto-detection from browser locale
  • Translation management UI

Technical Feasibility

Feasible

Build-on: React app already uses component-based architecture. String extraction is mechanical.

Build-new: i18n provider, locale file loading, language picker component, string extraction tooling.

Risks: String extraction is labor-intensive but low-risk. Pluralization rules differ across locales (Japanese has no plural forms; Spanish has two). ICU MessageFormat handles this.

Sequencing: Framework first, then extract strings, then translate. Translations can be parallelized.

Resolved Questions

1
Should we ship RTL support in this phase?
No — layout foundation only (CSS logical properties). Full RTL (Arabic, Hebrew) deferred to a follow-up. Rationale: Japan and LATAM are the priority markets; neither requires RTL.
2
ICU MessageFormat vs. simpler key-value approach?
ICU MessageFormat. It handles pluralization, gender, and number formatting natively. The complexity cost is upfront (framework setup) but saves rework when adding languages later.

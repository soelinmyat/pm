! Problem & Context

Team leads wait 2-3 days for filtered views that should be self-service.

They rely on engineering to create custom SQL queries, consuming ~15% of engineering support time. This is a self-service gap, not a missing feature.

Research signal: 8 of 12 team leads interviewed have requested filtered views in the past quarter. Both FilterCo and DashPro offer self-service filtering, making this a switching motivator for prospects evaluating alternatives.

▢ Scope Overview

Five filter capabilities in scope; custom metrics and export deferred.

✓ In Scope

  • Date range filter (preset ranges + custom)
  • Team/department filter (multi-select)
  • Metric type filter (revenue, usage, engagement)
  • Filter persistence across sessions
  • Shareable filtered views via URL params

✗ Out of Scope

  • Custom calculated metrics (deferred — requires formula engine)
  • Export filtered data to CSV (Phase 2 candidate)
  • Scheduled filter reports (low demand signal)
  • Filter-level access control (revisit post-launch)
⚡ 10x — single-step filtering vs competitors' 3-screen flow

⚖ Competitive Context

No competitor offers single-step filtering with URL sharing.

Capability FilterCo DashPro Our Approach
Filter application 3-screen flow Sidebar + Apply click Inline, instant apply
URL sharing Not supported Manual share button Auto URL encoding
Session persistence Login required Cookie, clears on logout localStorage + URL
Custom date ranges Presets only Date picker only Presets + picker
Key differentiator: Single-step filtering + URL sharing. Both competitors lack shareable filter links.

⚙ Technical Feasibility

Feasible as scoped — builds on existing API and component patterns.

Verdict: ✓ Feasible as scoped

▶ Build On

  • src/api/dashboard.ts — existing endpoints, add query param support
  • src/components/Dashboard/ — widget rendering, already accepts data props
  • src/middleware/auth.ts — auth pattern, reuse as-is
  • src/hooks/useQueryParams.ts — existing URL param utility

✚ Build New

  • Filter bar component (new, ~300 lines estimated)
  • Filter query engine on API side (extend existing endpoint)
  • localStorage sync hook (small utility)
  • Date picker integration (use existing design system component)

⚠ Risks

  • Query performance on large datasets — need indexing on date + team columns
  • URL param encoding for complex filter states — test with edge cases

→ Sequencing

  • 1. API query engine — everything depends on this
  • 2. Filter bar component — UI shell
  • 3. Date range filter — most complex UI control
  • 4. Persistence + URL sharing — polish layer

? Resolved Questions

Questions raised during review, with answers.

All reviewer questions resolved using research data and strategy context.

1
What is the expected reduction in engineering support requests?
Currently ~15% of support time goes to custom query requests. Based on FilterCo's published case studies, self-service filtering reduces ad-hoc query requests by 40-60%. Conservative target: 50% reduction in 90 days.
2
Why prioritize this over the CSV export feature that was also requested?
Filtering enables self-service for the common case (8 of 12 team leads). Export is a power-user feature with 3 requests total. Research signal is 2.7x stronger for filtering. Export remains a Phase 2 candidate.
3
How do we measure success in 90 days?
70%+ of active team leads using filters weekly (tracked via existing analytics), and engineering custom query requests drop by 50% (measured from support ticket tags).

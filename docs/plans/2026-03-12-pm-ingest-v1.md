# PM Plugin: `/pm:ingest` v1

## Summary

`/pm:ingest` is the customer-evidence entry point for PM.

It accepts a file path or folder path, imports raw customer evidence, normalizes it into a private machine-readable store, and updates durable research artifacts in the shared `pm/` knowledge base.

This is a single user-facing skill with two internal phases:

1. Import and normalize evidence
2. Synthesize research artifacts into `pm/research/`

The user should think: "I have customer evidence. Ingest it."

Not: "First I ingest, then I run a second synthesis command."

---

## Goal

Expand PM from mostly external market research into customer evidence.

Today PM is strongest at:
- market landscape
- competitor profiling
- strategy synthesis
- grooming ideas into issues

What is missing is customer evidence:
- support tickets
- interview transcripts
- sales call notes
- feature request exports
- churn reasons

`/pm:ingest` closes that gap.

---

## Command Surface

### Primary command

```text
/pm:ingest <path>
```

Examples:

```text
/pm:ingest ~/Downloads/interviews/
/pm:ingest ~/Desktop/support-export.csv
/pm:ingest ./research/customer-calls/
```

### No-arg behavior

```text
/pm:ingest
```

If no path is provided:
- If prior imports exist, ask:
  - "Do you want to refresh research from existing imported evidence, or ingest a new file/folder path?"
- If no prior imports exist, ask:
  - "Provide a file or folder path containing customer evidence to ingest."

### User promise

`/pm:ingest` should:
- accept evidence wherever it already lives
- avoid requiring users to move files into a special folder
- preserve privacy by storing normalized evidence in `.pm/`
- update durable summaries in `pm/research/`

---

## Inputs

### Supported path types

- Single file path
- Directory path

If a directory is provided:
- recursively scan supported files
- ignore hidden files and system artifacts by default
- report skipped files

### Supported file types in v1

- `.md`
- `.txt`
- `.csv`
- `.json`

### Deferred from v1

- `.pdf`
- `.docx`
- direct cloud URLs
- live integrations
- email inbox sync
- call recording audio

These can come later once the canonical evidence model is proven.

## User Intents

`/pm:ingest` should handle these three intents:

1. "Import these interview notes"
2. "Import this support export and tell me the patterns"
3. "Refresh our customer research after adding more evidence"

The skill is not just a parser. It is an evidence-to-insight workflow.

## Output Model

### Private normalized storage

All normalized records live in `.pm/`.

Recommended structure:

```text
.pm/
  config.json
  sessions/
  imports/
    manifest.json
  evidence/
    source-0001.json
    source-0002.json
```

Rules:
- `.pm/` stays gitignored
- raw source files remain at their original path
- normalized records are copied into `.pm/evidence/`
- import metadata lives in `.pm/imports/manifest.json`

### Durable shared outputs

Customer evidence summaries live in committed `pm/`.

Recommended structure:

```text
pm/
  research/
    index.md
    bulk-editing/
      findings.md
    onboarding-friction/
      findings.md
    reporting-gaps/
      findings.md
```

Rules:
- `pm/research/` is committed
- these files should be safe to share
- preserve provenance in frontmatter without copying raw private data
- PII handling: the skill must **warn the user to review** findings before committing rather than silently promising automatic redaction — Claude's PII detection is not reliable enough to guarantee safe output

### Coexistence with `pm:research`

`pm:research` already creates `pm/research/index.md` and `pm/research/{topic}/findings.md` for external market research (web searches, competitor patterns). Customer evidence from `/pm:ingest` writes to the same directory.

Coexistence rules:
- `pm/research/index.md` is a single unified index — both skills append to it
- Each entry in the index carries a `Source Origin` column: `external` (from pm:research) or `internal` (from pm:ingest)
- Each `findings.md` carries `source_origin` in frontmatter for the same purpose
- Neither skill overwrites the other's entries — append and update only your own
- The dashboard Topics tab shows both origins with visual distinction

## Canonical Evidence Model

Each normalized record should map into a common schema regardless of source format. The schema is split into **core** (required — reliable to extract) and **optional** (best-effort — populated when the source provides clear signal).

### Normalized record

```json
{
  // --- Core (required) ---
  "id": "uuid-or-stable-hash",
  "source_path": "/absolute/path/to/file",
  "source_type": "interview|support|sales|notes|feedback|unknown",
  "source_format": "md|txt|csv|json",
  "imported_at": "2026-03-12T10:00:00Z",
  "topic": "bulk editing",
  "pain_point": "editing many rows is slow",
  "summary": "Customer requested bulk edit capability for recurring workflows.",
  "quote": "Editing 50 rows one by one is painful.",
  "raw_ref": {
    "file": "/absolute/path/to/file",
    "row": 14,
    "section": "Pain points"
  },

  // --- Optional (best-effort) ---
  "event_date": "2026-03-10",
  "title": "Customer interview with ops lead",
  "account": "redacted-or-null",
  "segment": "SMB|mid-market|enterprise|null",
  "persona": "ops-manager|null",
  "requested_outcome": "bulk actions",
  "severity": "low|medium|high|null",
  "confidence": "low|medium|high",
  "tags": ["workflow", "bulk-actions"]
}
```

### Notes

- Core fields must always be populated — if `topic` or `pain_point` cannot be extracted, the record is too ambiguous to ingest and should be reported as a parse warning
- Optional fields are populated when the source provides clear signal — do not hallucinate values for sparse records
- `source_type` can be inferred from filename, columns, or user prompt — when confidence is low, ask the user (see Skill Flow Phase 1)
- `topic`, `pain_point`, and `requested_outcome` are extracted fields, not raw input
- `raw_ref` exists for local traceability — it uses absolute paths (kept private in `.pm/evidence/`). When writing committed `findings.md`, convert to portable labels (filename + row/section)

## Import Manifest

`/pm:ingest` must be idempotent enough for repeated use.

Use `.pm/imports/manifest.json` to track:

```json
{
  "version": 1,
  "last_synthesis_at": "2026-03-12T10:05:00Z",
  "imports": [
    {
      "path": "/absolute/path/to/support-export.csv",
      "kind": "file",
      "sha256": "abc123",
      "imported_at": "2026-03-12T10:00:00Z",
      "record_count": 143,
      "format_hint": "csv-support",
      "column_mapping": {
        "title": "Subject",
        "description": "Body",
        "priority": "Severity",
        "date": "Created At"
      }
    }
  ]
}
```

Behavior:

**Unchanged files** (hash matches prior import):
- Skip by default
- Tell the user it was already imported

**Modified files** (same path, different hash):
- Detect via path match + hash mismatch in manifest
- Ask the user: "support-export.csv has changed since last import. Re-import and replace 143 existing records?"
- On confirmation: delete old evidence records for that file, re-import, update manifest entry with new hash and timestamp
- Synthesis re-runs for affected themes (records with matching `source_path`)

**Deleted/moved files** (path in manifest no longer exists):
- Detected during no-arg refresh or explicit re-import
- Report to user: "2 previously imported files no longer exist at their original path"
- Do NOT automatically delete evidence records — the evidence is still valid even if the source moved
- Offer: "Remove orphaned records? (y/n)" — user decides

**Explicit re-import:**
- User can force re-import of a previously imported file
- Old records for that file are replaced, not duplicated

## Parsing Rules

### Markdown / text

Expected sources:
- interview notes
- call summaries
- founder notes
- pasted support conversations

Behavior:
- split by headings, bullets, or paragraphs
- identify likely pain points, requests, complaints, and segment cues
- extract notable quotes where available

### CSV

Expected sources:
- support exports
- feature request logs
- sales call trackers

Behavior:
- detect headers
- map common columns like `title`, `description`, `account`, `priority`, `segment`, `date`
- **show detected column mapping to the user and ask for confirmation** before importing — CSV schemas vary wildly and a `priority` column could mean severity, business priority, or sort order
- on confirmation, classify each row into normalized evidence
- cache the confirmed mapping in the manifest (`column_mapping` field) so re-imports of the same schema skip confirmation

### JSON

Expected sources:
- tool exports
- transcript bundles
- support API dumps

Behavior:
- parse arrays or top-level objects
- map known keys when possible
- fall back to generic extraction if schema is unknown

### Unknown structures

If parsing is incomplete:
- ingest what is reliable
- report ambiguous files
- never silently invent structured data

## Research Synthesis

After normalization, `/pm:ingest` synthesizes customer evidence into research artifacts.

### What a theme is

A theme is a **problem cluster** — a group of related pain points that point to the same underlying user need or outcome gap. One theme may contain multiple surface-level complaints that share a root cause.

Granularity rule: cluster by the **outcome the user wants**, not by surface-level keyword. "Topic" is too broad (everything about onboarding), individual "pain point" is too narrow (splits related signals). Problem cluster is the right level.

Examples:
- **Bulk editing** — users report slow row-by-row editing, missing multi-select, and no batch operations → same outcome gap: "I need to act on many items at once"
- **Onboarding confusion** — SMB accounts report unclear setup steps, missing defaults, and no guided tour → same outcome gap: "I need to get productive without reading docs"
- **Reporting gaps** — enterprise accounts request custom reports, export formats, and scheduled delivery → same outcome gap: "I need to prove ROI to my stakeholders"
- **Integration fragility** — churn reasons cite broken syncs, missing webhooks, and stale data → same outcome gap: "I need my tools to stay connected reliably"

### Synthesis rules

Cluster records by:
- topic
- pain point
- requested outcome
- persona
- segment

Score themes by:
- frequency
- severity
- recency
- segment concentration
- strategic relevance to `pm/strategy.md` if it exists

### v1 output format

Both `pm:research` (external) and `pm:ingest` (internal) write to the same `pm/research/` directory using a **unified schema**. This section defines the canonical contract that both skills must follow.

> **Implementation note:** The current `pm:research` SKILL.md (Topic Mode, lines 168-194) uses a simpler frontmatter schema without `source_origin`, `evidence_count`, or `segments`. That skill must be updated to match this unified schema when `/pm:ingest` ships. Until then, the existing format remains valid — `source_origin` defaults to `external` when absent.

#### `pm/research/index.md` (unified)

```markdown
---
type: research-index
created: YYYY-MM-DD
updated: YYYY-MM-DD
---

# Research Index

| Topic | Origin | Evidence | Summary | Updated |
|---|---|---:|---|---|
| Bulk editing | internal | 17 records | Users need batch operations for repetitive workflows | 2026-03-12 |
| AI pricing models | external | 3 sources | Dynamic pricing is table-stakes but underserved | 2026-03-11 |
| Onboarding friction | mixed | 8 records, 2 sources | SMB accounts struggle without guided setup | 2026-03-12 |
```

Column definitions:
- **Origin:** `internal` (from pm:ingest), `external` (from pm:research), `mixed` (both)
- **Evidence:** record count for internal, source count for external, both for mixed
- **Summary:** one-line — what did we learn?

Rules:
- Both skills append to and update this table — neither overwrites the other's rows
- A topic becomes `mixed` when both skills contribute to the same slug
- Each row links to `pm/research/{slug}/findings.md`

### Multi-writer ownership for mixed-origin files

When both `pm:research` and `pm:ingest` contribute to the same topic slug, the file becomes `mixed`. To avoid destructive overwrites, each skill owns specific sections:

**Frontmatter ownership:**
- `source_origin`: set to `mixed` by whichever skill writes second
- `sources` array: each skill appends its own entries (url-based for external, file-based for internal) — never removes the other's
- `evidence_count`, `segments`, `confidence`: owned by `pm:ingest` — `pm:research` leaves these untouched
- `topic`, `created`, `updated`: shared — either skill may update `updated`

**Body section ownership:**
- **Summary**: last writer wins — the skill that runs most recently rewrites it to incorporate both internal and external findings
- **Findings**: each skill appends its own numbered findings, prefixed with origin label: `[external]` or `[internal]`. Neither skill deletes the other's findings.
- **Representative Quotes**: owned by `pm:ingest` — `pm:research` does not modify
- **Strategic Relevance**: last writer wins (both skills may update)
- **Implications**: last writer wins (both skills may update)
- **Open Questions**: additive — both skills append, neither removes
- **Source References**: each skill appends its own references — neither removes the other's

**Write protocol:**
1. Read the existing file
2. Parse frontmatter and body sections
3. Update only your owned sections
4. Merge shared sections (Summary, Strategic Relevance, Implications) by rewriting to incorporate both sources
5. Write the file back

This means neither skill needs to understand the other's internal logic — just respect section boundaries.

#### `pm/research/{theme-slug}/findings.md` (unified)

```markdown
---
type: topic-research
topic: Bulk Editing
created: YYYY-MM-DD
updated: YYYY-MM-DD
source_origin: internal|external|mixed
sources:
  # External sources (from pm:research)
  - url: https://example.com/article
    accessed: YYYY-MM-DD
  # Internal sources (from pm:ingest) — portable labels, not absolute paths
  - label: support-export.csv
    rows: [12, 14, 31]
    imported: YYYY-MM-DD
  - label: interview-ops-lead.md
    section: Pain points
    imported: YYYY-MM-DD
    accessed: YYYY-MM-DD
# --- Fields below are optional, present when source_origin includes internal ---
evidence_count: 17
segments:
  - SMB
  - Mid-market
confidence: high
---

# Bulk Editing

## Summary
2-3 sentences on what this theme is and why it matters.

## Findings
Numbered findings with supporting evidence and source references.

## Representative Quotes
> "Editing 50 rows one by one is painful."

## Strategic Relevance
How this supports or challenges the current strategy.
If inferred, label it: *Inferred from evidence patterns, not directly stated.*

## Implications
What this means for the product. Link to strategy sections if relevant.

## Open Questions
What this research did NOT answer.

## Source References
- support-export.csv (rows 12, 14, 31) — imported 2026-03-12
- interview-ops-lead.md (section: Pain points) — imported 2026-03-10
- https://example.com/article — accessed 2026-03-11
```

The unified body sections are:
- **Summary** — always present
- **Findings** — always present
- **Representative Quotes** — present when internal evidence provides quotes
- **Strategic Relevance** — present when `pm/strategy.md` exists
- **Implications** — always present
- **Open Questions** — always present
- **Source References** — always present, with portable labels for internal files and URLs for external

Rules:
- no raw private dumps in committed files
- source references in committed files must use **portable labels** (filename + row/section metadata), not absolute local paths — absolute paths are machine-specific and useless to collaborators
- absolute paths are kept only in `.pm/imports/manifest.json` (gitignored) for local traceability
- representative quotes should be flagged for user review before committing — do not assume automatic PII redaction is sufficient
- if strategic relevance is inferred, label it clearly as inference
- provenance is carried in frontmatter `sources` array and body Source References section

## Skill Flow

### Phase 1: Intake

1. Accept the provided path or ask for one
2. Determine whether the path is a file or folder
3. Preview file count and formats
4. Infer source type from filename, columns, or content — if confidence is low, ask the user: "This looks like support tickets — correct?"
5. For CSV files, show detected column mapping and ask for confirmation
6. Ask for confirmation if:
   - file count is very large
   - unsupported formats dominate
   - re-import would duplicate many files
   - source type or column mapping is ambiguous

### Phase 2: Import

1. Parse each supported file
2. Normalize evidence records
3. Write normalized records to `.pm/evidence/`
4. Update `.pm/imports/manifest.json`
5. Report:
   - files scanned
   - files imported
   - files skipped
   - records created
   - parse warnings

### Phase 3: Synthesize

Synthesis is **incremental** — only process changed records to avoid re-reading the entire evidence store on every run.

1. Read `.pm/imports/manifest.json` to find `last_synthesis_at` timestamp
2. Determine what changed since last synthesis:
   - **New records:** `imported_at` after `last_synthesis_at`
   - **Replaced records:** records whose `source_path` matches a re-imported file (hash changed)
   - **Removed records:** records flagged for removal by user (orphaned file cleanup)
3. For new/replaced records: group into problem clusters (themes)
4. For existing themes affected by changes: re-synthesize `findings.md` — update counts, quotes, findings, and source references
5. For themes whose records were all removed: delete the theme's `findings.md` and remove from index
6. For new themes: create new `pm/research/{theme}/findings.md`
7. Update `pm/research/index.md` — append new themes, update counts for modified ones (never overwrite `pm:research` external entries)
8. Write `last_synthesis_at` to manifest

### Phase 4: Summarize to user

Example end-state:

> "Imported 36 files and created 214 normalized evidence records.
> Updated 5 research themes in `pm/research/`.
> Strongest themes: bulk editing, reporting gaps, onboarding confusion."

## Downstream Consumption

`/pm:ingest` is only valuable if the rest of PM uses its outputs.

### `pm:dig`

Before web search, check:
- `pm/research/index.md`
- relevant `pm/research/{topic}/findings.md`

Use these to answer:
- whether a request is repeated
- which segments care
- how severe the pain appears

### `pm:strategy`

Use customer evidence research to strengthen:
- ICP definition
- segment prioritization
- current priorities
- explicit non-goals

Usage should be explicit:
- "This priority is supported by research theme X"
- "This proposed segment expansion is contradicted by imported evidence"

### `pm:research`

Combine:
- external market evidence
- competitor patterns
- customer evidence from `pm/research/`

This is important because product strategy should not be derived from competitor behavior alone.

### `pm:groom`

When drafting issues, include:
- relevant research links
- evidence count
- representative quote if useful
- affected segments/personas

Each issue should be able to answer:
- who asked for this
- how often this appears
- how painful it seems
- whether it supports current strategy

### `pm:view` (dashboard)

Customer evidence themes appear in the Research page's Topics tab alongside external research topics. Visual distinction:
- Topics with `source_origin: internal` show a badge like "12 evidence records"
- Topics with `source_origin: external` show the existing style
- Topics with `source_origin: mixed` (both internal evidence and external research) show both indicators

## Product Decisions

### Why path-based instead of inbox-first

Because users already have exports where they are. Moving files adds friction.

### Why one skill instead of separate import and synthesis commands

Because users think in terms of outcome, not pipeline stages.

### Why `.pm/` for normalized data

Because normalized evidence is machine storage, private, and likely contains sensitive details.

### Why `pm/research/` for outputs

Because customer evidence and market evidence should live in one human-facing knowledge base, with provenance carried by frontmatter rather than folder sprawl.

---

## Resolved Questions

1. **Source type labeling:** Yes — ask the user when confidence is low. A one-line prompt ("This looks like support tickets — correct?") is cheap and prevents cascading misclassification.
2. **PII redaction:** Warn, don't auto-redact. The skill flags findings for user review before committing. A false sense of security is worse than no redaction.
3. **Theme granularity:** One theme = one problem cluster. Cluster by the outcome the user wants, not by surface-level keyword. See "What a theme is" section.
4. **Weak signal artifacts:** Not in v1. Weak signals simply don't appear as themes. A "dismissed" artifact adds maintenance burden before the signal quality baseline is established.

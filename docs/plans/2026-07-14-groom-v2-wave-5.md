# Groom v2 and proposal quality — Wave 5

## Outcome

Make `pm:groom` a provider-neutral, resumable product-definition lifecycle whose canonical output is one structured proposal source. Generate the human HTML reader and downstream RFC/Dev handoff from that source, bind product approval to exact proposal bytes, and measure proposal usefulness rather than instruction presence.

The workflow must run well on GPT-5.6 Sol High and Opus 4.8 xHigh without embedding provider-specific coaching or turning quick grooming into full ceremony.

## Baseline evidence

The v1.13.25 baseline passes plugin validation and 61 focused Groom/artifact tests with two dependency skips. Both installed workhorse caches pass plugin validation.

The baseline also demonstrates the migration need:

- Groom state is an unvalidated Markdown frontmatter document that steps edit directly.
- Draft and Present ask the model to maintain Markdown and HTML twins by hand.
- Proposal content has no canonical machine schema or exact approval audit.
- RFC intake checks proposal prose/status rather than consuming a structured approved contract.
- Fixed reviewer/persona references generate unresolved-persona warnings in step-loader tests.
- Most Groom regression tests assert headings and phrases, not state transitions, artifact identity, approval safety, or output quality.

## Product decisions

### One source, three projections

The canonical proposal is `{pm_dir}/backlog/proposals/{slug}.json`.

It produces:

1. `{pm_dir}/backlog/proposals/{slug}.html` — primary human reader.
2. `{pm_dir}/backlog/{slug}.md` — compatibility/read-model projection for backlog, List, Board, sync, and existing project conventions.
3. A bounded execution-contract view consumed directly by RFC and Dev.

The generated Markdown is not an independently editable content source. Legacy Markdown-only proposals remain readable and receive an explicit migration path; no existing project must migrate merely to list or inspect old work.

### Approval is a separate fact

Drafting and team review may establish `reviewed`, but only an explicit user action establishes `approved`. Approval writes `{pm_dir}/backlog/proposals/{slug}.approval.json`, bound to the canonical proposal hash, lifecycle revision, approver identity, and timestamp. Any substantive revision invalidates prior review and approval.

Generated Markdown lifecycle remains compatible with existing states:

- canonical `draft` / `reviewed` / `approved` project to backlog `drafted` / `drafted` / `proposed`;
- RFC approval still advances the backlog projection to `planned`;
- Dev and Ship retain `in-progress` and `done`.

### Questions, not reviewer headcount

Review work is derived from independent decision questions. A route packet contains the current proposal identity plus questions such as:

- Is the problem/evidence chain sufficient for this decision?
- Is the scope coherent, minimal, and explicit about non-goals?
- Are acceptance criteria observable and implementation-neutral?
- Are user flows, failure states, and design requirements complete?
- Is feasibility credible without smuggling in an engineering design?
- What assumption, counterexample, or competitive fact could reverse the recommendation?

The runtime may answer questions inline or distribute them across available workers. Correctness depends on question coverage and evidence-bound synthesis, never an exact worker count or persona name.

### Tier controls depth

- `quick`: intake, bounded research assessment, scope, draft, explicit approval, handoff.
- `standard`: adds strategy, full research, synthesis, design requirements, and independent review questions.
- `full`: adds broader question coverage, presentation quality, and rendered Design Critique when proposal UI/template material changes.
- `agent`: retained as a compatibility alias during this wave, routed through the same provider-neutral full contract with stricter KB gates; the Claude-only refusal is removed.

Tier changes depth and question coverage, not source integrity, approval binding, or artifact validation.

## Architecture

### Groom session runner

Add `scripts/groom-session.js` backed by `scripts/lib/groom-session-schema.js` and the shared workflow runtime primitives.

Canonical private state:

```text
.pm/groom-sessions/{slug}/session.json
```

Phases:

```text
intake → research → scope → synthesis → design → draft → review → presentation → approval → handoff → retro
```

Tier routing may omit inapplicable phases, but every completed phase records a strict result envelope, evidence, transition history, runtime identity, and exact current proposal revision when one exists.

Required runner operations:

- `init`, `status`, `next`, `context`, `record`, `revise`, `approve`, `approval-audit`, `validate`, and `migrate`;
- idempotent terminal retry and immutable run-ID archive;
- provider-neutral model profile and prompt packet;
- explicit authority for tracker or other external effects;
- migration from current `.pm/groom-sessions/{slug}.md` state.

### Proposal schema and generator

Add:

- `scripts/proposal-check.js` — strict schema, semantic, source/HTML/hash, projection, lifecycle, and approval validation;
- `scripts/proposal-render.js` — atomic JSON-to-HTML and JSON-to-Markdown generation using the proposal template contract;
- `scripts/groom-prompt.js` — phase-local bounded prompt packets for proposal work;
- `scripts/lib/proposal-schema.js` — pure validation, normalization, projection, and content-hash logic.

Schema version 1 includes:

- identity, lifecycle, revision, title, outcome, priority, size, labels, and source lineage;
- decision brief and recommendation;
- audience, JTBD, evidence refs, assumptions, and confidence;
- structured scope, non-goals, requirements, acceptance criteria, edge cases, design requirements, and success metrics;
- alternatives, risks, open decisions, and resolved decisions;
- question-review outcomes and advisory debt;
- presentation metadata and downstream handoff fields.

Unknown fields fail closed. IDs are stable. Required arrays reject empty or duplicate entries where absence would make the handoff non-executable. Paths and citations remain project-bounded.

### RFC and Dev consumption

RFC intake prefers the approved proposal JSON plus approval audit. It verifies hash/revision/status and reads scope, acceptance criteria, non-goals, design requirements, edge cases, and open decisions directly. It falls back to legacy Markdown proposals only through a named compatibility path.

Dev readiness uses the same proposal reader. It must not treat a generated Markdown status edit as product approval when a canonical JSON proposal exists.

### Reader artifact

Preserve the established visual language while making generation deterministic:

- offline, inert, accessible, responsive, printable HTML;
- decision brief and execution contract before appendix detail;
- visible lifecycle and approval state;
- source, revision, content hash, evidence freshness, and open-decision markers;
- no hidden active scripts, network dependencies, or model-authored hash repair;
- retained artifact and render manifests for material template changes.

### W5.5 — Downstream handoff

Owns RFC/Dev proposal readers, approval verification, compatibility fallback, command/docs parity, and integration tests.

## Acceptance criteria

1. A fresh or migrated Groom session has deterministic routing, strict results, durable transition history, and idempotent resume.
2. Quick, standard, full, and agent-compatible tiers share one integrity/approval contract while retaining proportionate depth.
3. GPT-5.6 Sol High and Opus 4.8 xHigh receive the same provider-neutral prompt/result contract; capability downgrades are explicit records.
4. One canonical JSON proposal atomically generates synchronized HTML and Markdown projections.
5. Proposal approval is explicit, hash-bound, revision-bound, independently auditable, and invalidated by substantive edits.
6. RFC and Dev prefer the approved structured proposal and cannot be fooled by prose/status drift in generated Markdown.
7. Legacy Markdown proposals and legacy Groom sessions remain readable through bounded, tested compatibility paths.
8. Review covers independent product questions and evidence; no contract depends on a fixed reviewer count or unresolved persona reference.
9. The proposal checker rejects malformed identity, unsafe paths/content, broken source bindings, projection drift, stale approval, and missing executable handoff fields.
10. Material proposal-template changes pass artifact validation, retained multi-viewport/print rendering, and current Design Critique.
11. Strong and weak schema-valid proposal fixtures are behaviorally eligible, and blind scoring separates them by a material committed threshold.
12. Core Groom cases cover happy path, ambiguity, resume, blocked/recovery, low-quality schema-valid output, and repeated-run variance.
13. Focused suites, full plugin tests, plugin validation, and both installed-cache canaries pass at the exact release commit.

## Non-goals

- A generic workflow engine or shared product-policy layer.
- Automatic product approval, tracker creation, RFC approval, or implementation.
- Requiring HTML for lightweight research or conversational thinking artifacts.
- Preserving arbitrary manual edits to generated proposal projections.
- Treating live-model output as deterministic CI.

## Test strategy

Start with failing contract tests for session creation/routing, proposal validation, generation determinism, approval invalidation, and RFC/Dev preference. Add malicious/path-boundary and crash-window cases around atomic writes. Keep generator golden fixtures semantic where possible; avoid enormous byte snapshots.

Quality certification uses blinded strong/weak fixtures with identical schema validity and task identity. Behavioral eligibility precedes quality scoring. Live Sol/Opus runs are release canaries and may be skipped with an explicit capability/availability receipt rather than blocking deterministic CI.

## Release gates

1. Focused Groom/proposal/session/handoff suites.
2. Plugin authoring and contract validation.
3. Full test suite.
4. Patch version prepared before final evidence freeze.
5. Design Critique for the changed proposal reader/template.
6. Evidence-bound Review with at most three remediation rounds in one lineage.
7. Both installed vNext caches validate and pass representative Groom/RFC/Dev smoke suites.
8. Hosted CI, squash merge, and release tag on the main merge commit.

## Done-when

All acceptance criteria are evidenced at one release commit, the release PR is merged, its tag points to main, the Wave 5 Dev session is complete, and the master plan release map identifies Wave 6 as next.

## Implementation evidence (pre-certification)

- Groom uses one provider-neutral 11-step sequence with deterministic tier routing and no agent-only duplicate steps.
- Canonical proposal JSON has strict closed-schema validation, semantic revision hashing, exact-byte approval audits, generated HTML/Markdown projections, and bounded legacy reads.
- Approval trust survives only monotonic `approved → planned → in-progress → done` lifecycle changes; semantic, revision, and decision drift fail closed.
- RFC and Dev intake derive size and acceptance criteria from the approved execution contract and reject contradictory caller input.
- The committed strong and weak fixtures are both schema-valid; deterministic blind calibration currently scores them 100/100 and 15/100.
- The focused Groom/proposal/RFC/Dev suite passes 129 tests, and plugin validation passes all 21 rules with no issues.

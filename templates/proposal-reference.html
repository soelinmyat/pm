Dashboard Filtering System

Users can narrow dashboard data to their team's metrics without requesting custom queries from engineering.

5 issues (1 parent + 4 children) 2 team review rounds 1 bar raiser round 2026-03-16

Problem & Context

Team leads currently rely on engineering to create custom SQL queries when they need filtered views of dashboard data. This creates a 2-3 day turnaround for what should be a self-service operation, consuming ~15% of engineering support time.

User interviews (pm/research/dashboard-filters/findings.md) confirmed that 8 of 12 team leads have requested filtered views in the past quarter. Competitor analysis shows that FilterCo and DashPro both offer self-service filtering, making this a switching motivator for prospects evaluating alternatives.

Scope Overview

In Scope

  • Date range filter (preset ranges + custom)
  • Team/department filter (multi-select)
  • Metric type filter (revenue, usage, engagement)
  • Filter persistence across sessions
  • Shareable filtered views via URL params

Out of Scope

  • Custom calculated metrics (deferred — requires formula engine)
  • Export filtered data to CSV (Phase 2 candidate)
  • Scheduled filter reports (low demand signal)
  • Filter-level access control (revisit post-launch with usage data)
10x — single-step filtering vs competitors' 3-screen flow

Issue Breakdown

PM-025 Dashboard Filtering System

Users can narrow dashboard data to their team's metrics without requesting custom queries from engineering.

feature dashboard high priority
  1. Dashboard renders a filter bar above the main content area with date, team, and metric type controls.
  2. Applying any filter updates all dashboard widgets within 2 seconds for datasets up to 100k rows.
  3. Active filters persist across browser sessions via localStorage, restored on next visit.
  4. Filtered state is encoded in URL query params; sharing the URL reproduces the exact filtered view.
  5. Clearing all filters restores the default unfiltered dashboard in under 500ms.
PM-026 Filter bar component and layout

Dashboard has a persistent, accessible filter bar that doesn't compete with data visibility.

frontend high priority
  1. Filter bar renders horizontally above dashboard widgets on screens >= 768px; collapses to a toggle drawer on mobile.
  2. Each filter control has a visible label, current selection indicator, and clear button.
  3. Filter bar consumes no more than 64px of vertical space in its default state.
  4. All filter controls are keyboard-navigable and announce state changes to screen readers.
PM-027 Filter query engine and API

Dashboard widgets receive filtered data from a performant query layer, replacing manual SQL requests to engineering.

backend high priority
  1. GET /api/dashboard?date_from=X&date_to=Y&teams=A,B&metrics=C returns filtered widget data.
  2. Query response time < 500ms for datasets up to 100k rows (p95, measured at API layer).
  3. Invalid filter combinations return a 400 response with a human-readable error message, not a 500.
  4. API follows existing auth pattern in src/middleware/auth.ts — no new auth mechanism.
PM-028 Filter persistence and URL sharing

Users don't lose their filter configuration between sessions, and can share exact views with teammates via URL.

frontend medium priority
  1. On filter change, URL query params update without a page reload (history.replaceState).
  2. On page load, filters initialize from URL params first; fall back to localStorage; fall back to defaults.
  3. localStorage key follows existing pattern: dashboard:filters:{userId}.
  4. Copying the URL and opening in a new tab reproduces the identical filtered view.
PM-029 Date range filter with presets

Users can filter by time period using one click for common ranges or a custom date picker for specific analysis.

frontend medium priority
  1. Preset options: Today, Last 7 days, Last 30 days, Last quarter, Year to date, Custom range.
  2. Custom range opens a date picker; selecting an invalid range (end before start) shows inline error.
  3. Selecting a preset visually highlights it; selecting custom deselects all presets.
  4. Default selection on first visit: Last 30 days.

Competitive Context

Capability FilterCo DashPro Our approach
Filter application 3 screens (select, configure, apply) Sidebar panel, requires "Apply" click Inline filter bar, instant apply on selection
URL sharing Not supported Manual "share" button generates link Automatic URL encoding, copy-paste ready

Key differentiator: single-step filtering with instant feedback vs competitors' multi-step workflows. URL-based sharing is a capability gap in both competitors that creates a meaningful collaboration advantage.

Technical Feasibility

Verdict: Feasible as scoped

Build On

  • src/api/dashboard.ts — existing dashboard data endpoints, add query param support
  • src/components/Dashboard/ — widget rendering, already accepts data props
  • src/middleware/auth.ts — auth pattern, reuse as-is
  • src/hooks/useQueryParams.ts — existing URL param utility

Build New

  • Filter bar component (new, ~300 lines estimated)
  • Filter query engine on API side (extend existing endpoint)
  • localStorage sync hook (small utility)
  • Date picker integration (use existing design system component)

Risks

  • Query performance on large datasets — need indexing on date + team columns
  • URL param encoding for complex filter states — test with edge cases

Sequencing

  • 1. API query engine (PM-027) — everything depends on this
  • 2. Filter bar component (PM-026) — UI shell
  • 3. Date range filter (PM-029) — most complex UI control
  • 4. Persistence + URL sharing (PM-028) — polish layer

Open Questions

Questions the decision-maker should be prepared to discuss:

  • What is the expected reduction in engineering support requests? (Currently ~15% of support time goes to custom query requests.)
  • Why prioritize this over the CSV export feature that was also requested? (Filtering enables self-service for the common case; export is a power-user feature with lower demand signal.)
  • How do we measure success in 90 days? (Suggested: 70%+ of active team leads using filters weekly, and engineering custom query requests drop by 50%.)
  • What is the migration path for existing saved custom queries? (Out of scope for v1 — existing queries continue to work, filters are additive.)
